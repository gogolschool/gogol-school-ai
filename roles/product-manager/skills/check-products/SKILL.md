# /check-products

## Что делает

Сверяет каждый активный/открытый набор (программу) между четырьмя источниками и сообщает, где они разъехались:

1. **Notion** — «🗂️ Таблица с продуктами» (она же «таблица наборов»), data source `collection://9728f62f-5c91-493b-8818-3bcdedae0a82`. Источник истины по плану набора (что менеджер по расписанию задумал). Статусы: «Проведено», «Согласовано», «В работе», «Отменено» — **активные/открытые = «Согласовано» + «В работе»**.
2. **Сайт** (gogolschool.ru / Bitrix, MCP `gogol-school-site`) — услуга-программа в iblock 16 (цена, активность, ссылка на оплату) и строка в расписании на `/schedule/` (HTML-список в `detail_text` элемента 27918 в iblock «Оферта» 38).
3. **OzmaDB** (MCP `ozma`) — продукт в `crm.actions`, мастера `crm.masters_for_actions` (→ `base.contacts`/`base.people`), дочерние занятия, `price`/`split_parts`.
4. **Google Calendar** (MCP `google-calendar`) — календарь **«GOGOL school»** (`gschool182@gmail.com`). Сверяем прежде всего **даты/время занятий** (события календаря против дат набора и дочерних занятий в Ozma). Календарь только для чтения — здесь ничего не правим.

Только **читает** и формирует отчёт + предлагает фиксы. Сам ничего не правит без явного подтверждения (см. «Правила фиксов»).

## Что сверяем

| Поле | Notion | OzmaDB | Сайт | Календарь (GOGOL school) |
|---|---|---|---|---|
| **Название** | «Название» / код в «Ссылка в CRM» | `crm.actions.name` | имя услуги (iblock 16) и текст `<li>` в `/schedule/` | заголовок события (`summary`) |
| **Мастера** | «Мастер-1»…«Мастер-4» | `crm.masters_for_actions.contact` (ФИО в `pun` ссылки) | (если упомянуты в `<li>` расписания) | (фамилии в `summary`) |
| **Даты** | «Первое занятие (дата)» / «Последнее занятие (дата)» | `action_datetime` / `end_datetime` ⚠️ **UTC** | даты в `<li>` расписания / `activeTo` услуги | дата/время события (`start`/`end`, TZ Europe/Moscow) |
| **Расписание** | «Расписание» | дочерние занятия (`crm.actions` где `parent_action=<id>`) | `<li>` в `/schedule/` | набор событий по дням (по одному на занятие) |
| **Активность/статус** | «Статус» | `is_deleted=false` / актуальность | `active=Y` + наличие `<li>` в `/schedule/` | событие не отменено (`status != cancelled`) |
| **Цены** | «Стоимость» | `price` (+ `split_parts`) | `price` (+ рассрочка в `prices`) | — |

Маппинг значений (справочники, формат дат/расписания, нумерация кодов) — **не дублируй здесь**, бери из мастер-промтов соседних скиллов: поля Notion → [add-product](https://www.notion.so/387612c762af81988be7c8edd61542ed), поля Ozma/сайта → [newproduct](https://www.notion.so/31f612c762af81ba999cc5657404528c) и [расписание](https://www.notion.so/33b612c762af816f99bef788a1ec8d28), календарь → [add-calendar](https://www.notion.so/387612c762af81cfb20ad67aaf962889).

## Ключи сопоставления

Джойн строим по полям, которые уже лежат в Notion-карточке:

- **Notion ↔ Ozma:** поле «Ссылка в CRM» (URL/ID продукта в `crm.actions`, напр. `…class_form?id=11856`). Пусто → продукт ещё не заведён в Озме (кандидат на `/newproduct`), не считай это расхождением полей, отметь отдельно.
- **Notion ↔ сайт:** поле «Ссылка на оплату» (URL вида `gogolschool.ru/payment/<id>/`, `<id>` = service_id в iblock 16). Пусто → ссылки на оплату ещё нет.
- **Сайт `/schedule/` ↔ набор:** по коду программы (`INT02-202608` и т.п.) в тексте `<li>`. В расписании живут только длительные программы (LAB/INT/LCOURSE/PI/HEAD) — для разовых МК/ОУ отсутствие в `/schedule/` это норма, не расхождение.
- **Календарь ↔ набор:** прямого ID нет — сопоставляй события по **названию/коду** (`summary` ≈ название/код набора) **и дате/времени**. Окно поиска бери по датам набора. Сначала отбери события набора, затем сверяй их даты/время с занятиями набора один-к-одному.

## Как работает

1. **Загрузи рабочий контекст из Notion** (рабочий MCP `c2755fd9-…`, не личный — см. память `feedback_notion_workspace_routing`): инструкцию [«Таблица по наборам»](https://www.notion.so/1d8612c762af80b4bfd6e609c68dff7b) и при необходимости мастер-промты `add-product`/`newproduct`/`add-calendar` для маппинга полей.
2. **Собери список наборов в охвате:** строки таблицы со статусом «Согласовано»/«В работе». ⚠️ SQL/view-запросы Notion MCP требуют платный план и **недоступны** — собирай наборы через `notion-search` (`data_source_url` = коллекция) и `notion-fetch` отдельных страниц (см. «Практические заметки»). Покажи пользователю, сколько наборов попадает в сверку, и дождись отмашки перед массовым проходом.
3. **Для каждого набора** подтяни четвёрку: карточку Notion, продукт Ozma (по «Ссылка в CRM»), услугу+расписание сайта (по «Ссылка на оплату» / коду) и события календаря «GOGOL school» в окне дат набора (`list-events`). Сверь поля из таблицы выше.
4. **Сформируй отчёт** (см. формат) и **предложи фиксы** по каждому расхождению с указанием, что и где править.
5. Правки выполняй только после подтверждения и только адресно (см. «Правила фиксов»).

## Что считается расхождением (и что — нет)

- **Время/даты:** OzmaDB хранит datetime в **UTC**, в Notion, на сайте и в календаре время местное (МСК = UTC+3, календарь в TZ Europe/Moscow). Перед сравнением приводи к одному поясу (`action_datetime` + 3 ч). Расхождение в ровно 3 часа = почти всегда артефакт пояса, а не реальная ошибка.
- **Календарь:** событие со `status = cancelled` = отменённое занятие — сверяй с активностью набора, а не считай «лишним». Заголовок первого занятия длительного продукта может содержать «ЗАПУСК» (конвенция `/add-calendar`) — это не другое событие, дата/время те же. Лишнее/пропущенное событие относительно занятий Ozma — расхождение.
- **Цены:** сравнивай как числа (нормализуй пробелы/«₽»/копейки). На сайте `prices` обычно содержит и рассрочку (напр. `1800×10`) — это не расхождение с базовой ценой.
- **Пустой ключ ≠ расхождение полей:** нет «Ссылка в CRM»/«Ссылка на оплату» — это «ещё не заведено», отдельная категория.
- **Разовые МК/ОУ вне `/schedule/`** — норма (туда идут только длительные программы).
- **Реальные расхождения:** разные мастера, разные даты (сверх пояса), разная цена, активен на сайте но удалён/закрыт в Озме (или наоборот), длительная программа открыта и на сайте, но отсутствует в `/schedule/`, занятие есть в Ozma но нет в календаре (и наоборот).

## Формат отчёта

Markdown, по образцу `/check-transactions` — кратко, по пунктам, без воды. Например:

```
## Сверка наборов на <дата>. Охват: N активных наборов

### ❌ Расхождения
1. **INT02-202608 «Есть контакт»** — нет в `/schedule/`, хотя интенсив, «Согласовано», на сайте активна, есть в Ozma и календаре. Фикс: добавить `<li>` (/newproduct → расписание).
2. **LAB02-202607 «…»** — цена: Notion 18 000 ₽, сайт 15 000 ₽, Озма 18 000 ₽. Фикс: …
3. **LAB02-202607 «…»** — в Озме 6 занятий, в календаре только 5 (нет 17.07). Фикс: добавить событие (/add-calendar).

### ⚠️ Не заведено
- **MK07-202606** — есть в таблице, «Ссылка в CRM» пустая → завести в Озме (/newproduct).

### ✅ Сходится: M наборов
```

В каждом фиксе указывай **источник истины и целевой источник** правки (что считаем верным и где меняем).

## Правила фиксов

- Режим — **отчёт + предложить фиксы**. Авто-правки не делаем. Для каждого расхождения предложи конкретное действие и **дождись подтверждения** по каждому (или явного «правь всё»).
- **Прод-безопасность.** Сайт (Bitrix) и Озма — продакшн. Правки в Озме — только через `transaction`, с предварительной сводкой. Правки на сайте — только адресно.
- ⚠️ **Расписание `/schedule/` перезаписывается целиком.** `iblock_element_update` затирает весь `detail_text` элемента 27918 (iblock 38). Сперва `iblock_element_get`, сохрани **весь** HTML, добавь/правь только нужный `<li>`. Если текст подозрительно короткий (нет ожидаемых сезонных разделов) — **СТОП**, не записывай, предупреди. Ничего не удалять без явной команды.
- **Календарь — read-only.** Скилл события не создаёт/не правит/не удаляет. Расхождения по календарю выноси в отчёт как задачу на правку через `/add-calendar`.
- Фактическую правку обычно делегируй профильному скиллу (`/newproduct` для Озмы/сайта, `/add-product` для Notion, `/add-calendar` для календаря) — так маппинг и самопроверка едины.

## Практические заметки (из обкатки)

- **Notion read-only через поиск.** `notion-query-data-sources` (SQL) и `notion-query-database-view` требуют Business/Enterprise + Notion AI — на текущем плане отдают 400. Наборы собирай через `notion-search` (`data_source_url` коллекции) и читай страницы `notion-fetch`. Имена ряда полей оканчиваются табом: `"Ссылка в CRM\t"`, `"Расписание\t"`, `"Первое занятие (дата)\t"` — учитывай при чтении.
- **`/schedule/` большой.** `iblock_element_get`(27918, 38) возвращает ~100k символов и сохраняется в файл, а не в вывод. Ищи код программы (`grep` по сохранённому файлу: `INT0\d-\d{6}`, тема, фамилия), не пытайся читать целиком.
- **Мастер из Ozma.** `crm.masters_for_actions.contact` — ссылка на `base.contacts`; ФИО берётся из `pun` ссылки (`{"id":20,"pun":"Карабань Сергей Викторович"}`). У `base.contacts` нет `first_name`/`last_name` — не джойни напрямую туда.
- **Календарь — аккаунт.** В MCP `google-calendar` ник аккаунта — `normal` (не `fabell1407`); календарь `gschool182@gmail.com`. Уточняй доступные аккаунты, если `list-events` вернёт «account not found».
