# /check-products

## Что делает

Сверяет каждый активный/открытый набор (программу) между четырьмя источниками — Notion «🗂️ Таблица с продуктами», сайт (Bitrix, iblock 16 + `/schedule/`), OzmaDB (`crm.actions`) и Google-календарь «GOGOL school» — и формирует Markdown-отчёт о расхождениях с предложенными фиксами. Только читает; правки — только после подтверждения.

## Как работает

1. **Загрузи актуальный мастер-промпт из Notion:** page `387612c762af81df9da6c72617610384`
   (https://www.notion.so/387612c762af81df9da6c72617610384). Действуй строго по нему — там четыре источника, что и как сверять, ключи сопоставления, что считать (и не считать) расхождением, формат отчёта, правила фиксов и практические заметки.
   > Notion берём через **рабочий** MCP (`c2755fd9-…`), не личный (см. память `feedback_notion_workspace_routing`).
2. Собери наборы в охвате (статус «Согласовано»/«В работе») через `notion-search` + `notion-fetch` (SQL/view-запросы Notion недоступны на текущем плане). Покажи число наборов и дождись отмашки.
3. Для каждого набора подтяни четвёрку (Notion / Ozma / сайт / календарь), сверь поля, сформируй отчёт и предложи фиксы.

## Критические правила

- **Только отчёт + предложение фиксов.** Авто-правок нет; по каждому расхождению дождись подтверждения.
- **Прод-безопасность.** Сайт (Bitrix) и Озма — продакшн: Озма — только через `transaction` со сводкой, сайт — адресно.
- ⚠️ **`/schedule/` перезаписывается целиком** (`detail_text` элемента 27918, iblock 38): сперва `iblock_element_get`, сохрани весь HTML, добавь только нужный `<li>`; короткий текст → СТОП.
- **Календарь — read-only**; расхождения по нему выноси как задачу на `/add-calendar`.
- Фактическую правку делегируй профильному скиллу (`/newproduct`, `/add-product`, `/add-calendar`).
