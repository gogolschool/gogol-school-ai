# /check-masters-workload

## Что делает

Проверяет смены мастеров за период на ошибки отметки, искажающие ЗП (неверный `masters`, дубли строк, пустой `masters`, непомеченный режиссёр, смены, отмеченные после выплаты, аномалии часов/ставок), сверяет «живой» расчёт с выплаченным снимком и выдаёт Markdown-отчёт с недоплатами/переплатами и фиксами. Только читает; правки — после подтверждения.

## Как работает

1. **Загрузи актуальный мастер-промпт из Notion:** page `38e612c762af8146b971d2f9bb56c038`
   (https://www.notion.so/38e612c762af8146b971d2f9bb56c038). Действуй строго по нему — там период, все проверки (A–M) с готовыми FunQL-запросами, правила интерпретации (что норма авто-split, а что дубль; учёт режиссёра), формат отчёта, правила фиксов и гочи FunQL. Кратко: A дубли · B masters · C пустой masters · D режиссёр · E правки после выплаты · F часы/ставки · G залочка повышенной ставки · H длительность смены (<15 мин) · I мастер на двух занятиях сразу · J домаркировка задним числом (жёлтый флаг) · K забытый «Ушёл» · L нет ставки → 0 · M продукт без фин.проекта/фин.направления.
   > Notion берём через **рабочий** MCP (`c2755fd9-…`), не личный (см. память `feedback_notion_workspace_routing`).
2. Спроси период (по умолчанию — прошлый месяц). Прогони проверки A–M по мастер-промпту.
3. Сформируй отчёт по мастерам с суммами недоплат/переплат и предложенными фиксами. Дождись подтверждения перед любой правкой.

## Критические правила

- **Только отчёт + предложение фиксов.** Авто-правок нет; по каждому пункту — подтверждение.
- **Прод-безопасность.** OzmaDB — продакшн: правки только через `mcp__ozma__transaction` со сводкой.
- **Не удалять записи** — дубль/ошибку обнулять `datetime`/`datetime_end` (по канон-инструкции).
- **Доплату — отдельной строкой** в `staffs.masters_salary_records` (`is_manual=true`, дата выплаты пустая) + транзакция `fin.economics`; нетить пере/недоплату одного мастера.
- **Повышенная ставка (Check G):** если база залочена не на ту ступень (пакетный ввод задним числом) — чинить выставлением `rate_manually = база(по реальному masters) + надбавка` и `high_rate_prev_manual = null`, `high_rate_applied` оставить `true`. До выплаты ЗП правка `rate_manually` подхватывается генерацией; после — недоплату отдельной строкой.
