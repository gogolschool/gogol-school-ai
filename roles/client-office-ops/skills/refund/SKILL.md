# /refund

## Что делает

Оформляет возврат по транзакции в Озме:

1. Идентифицирует исходную транзакцию (по `id`, `tks_order_id`, либо по ФИО покупателя + коду продукта). Если есть PDF-заявление — извлекает оттуда ФИО, продукт, сумму и дату.
2. Считает тип возврата:
   - **Полный** (`cancellation_status = 8`): сумма равна исходной либо удержана только банковская комиссия ≤ 3 % (типично 2.6 %).
   - **Частичный** (`cancellation_status = 9`): во всех остальных случаях.
3. Определяет канал возврата по `account_to` исходной транзакции (либо по формулировке в PDF):
   - `account_from = 6` (Tinkoff эквайринг) — если в заявлении «на карту, с которой произведена оплата» и исходный платёж был через Т-банк.
   - `account_from = 26728` (CloudPayments) или `23719` (Яндекс-сплит) — аналогично для других сервисов.
   - `account_from = 4434` (ООО, р/с Точка) — если в заявлении явные реквизиты стороннего банка или возврат через эквайринг не прошёл; деньги уходят отдельной банковской платёжкой.
4. Проверяет идемпотентность (`SELECT … WHERE linked_transaction = <id> AND NOT is_deleted`) — не дублирует уже сделанный возврат.
5. **Показывает оператору план и явно спрашивает подтверждение.** Не вставляет ничего до «да».
6. Создаёт транзакцию возврата в `fin.transactions` + обновляет `cancellation_status` на заявке `crm.actions_for_contacts` одной атомарной операцией.
7. Возвращает id, ссылки и (для возвратов с Точки) формулировку назначения платежа для банковской платёжки.

## Как работает

1. Загружает актуальный мастер-промпт из Notion: page `365612c762af811d8f91d0311da58e4a`.
2. Принимает от оператора входные данные: id транзакции, либо `tks_order_id`, либо ФИО + код продукта; сумму возврата (или PDF-заявление, откуда вытащить сумму и дату).
3. Находит исходную транзакцию и связанную заявку. При нескольких совпадениях — показывает варианты и просит выбрать.
4. Проверяет дубли возвратов на ту же `linked_transaction`.
5. Считает тип возврата (полный/частичный) и канал (Tinkoff/CP/Сплит/Точка).
6. Выводит сводку: исходная транзакция (id, сумма, продукт, заявка), сумма возврата + тип + обоснование, направление счетов, какой статус отмены проставится, для возвратов с Точки — текст назначения платежа.
7. Ждёт явного «да» — только после этого вызывает `mcp__OzmaMCP__transaction`.
8. Отчитывается: id новой транзакции возврата, новый статус заявки, при необходимости — id для поиска в ЛК эквайринга.

## Режим списка: что сейчас висит в статусе

Если оператор пишет `/refund список на возврат` (а также `/refund что подписано`, `/refund подписанные`, `/refund кто на возврате`) — это **read-only режим**: ничего не вставляем в Озму, просто отдаём выборку из Notion-таблицы возвратов.

- **Таблица**: `💸 Таблица возвратов`, data source `3d37e5a4-902d-4dd7-8f77-55d761fe42ba` (страница `03f3c1ed71b2462ebc69495b83353f4e`).
- **Какой статус по умолчанию**: оператор говорит «подписано», но в таблице точное значение поля `Статус` — **`подписано. положили в папку Бэлле`** (зелёный). Фильтруем именно по нему. Если оператор назвал другой статус («позвали на подпись», «согласовано», «подготовка заявления» и т.п.) — фильтруем по нему, сопоставив с реальными значениями поля `Статус`.
- **Что выводим** по каждой строке: `ФИО`, `Дата статуса`, `Сумма возврата`, `Название продукта в СРМ`, `Тип возврата`, `Откуда возврат`/`Оплата`. Сортировка по `Дата статуса` по возрастанию — кто дольше всех ждёт, тот сверху.

### Как достать строки

Таблица живёт **только в Notion** (этого статуса нет в Озме), поэтому читаем оттуда.

1. **Предпочтительно — `notion-query-data-sources` (SQL):** детерминированно и полно.
   ```sql
   SELECT "ФИО", "Дата статуса", "Сумма возврата", "Название продукта в СРМ",
          "Тип возврата", "Откуда возврат", "Оплата", url
   FROM "collection://3d37e5a4-902d-4dd7-8f77-55d761fe42ba"
   WHERE "Статус" = 'подписано. положили в папку Бэлле'
   ORDER BY "date:Дата статуса:start"
   ```
   ⚠️ Требует тариф **Notion Business + AI** — иначе MCP отдаёт `validation_error: This tool requires a Business plan`. На текущем тарифе **не работает** (проверено 29.06.2026).

2. **Фолбэк, работает на любом тарифе — чтение готовой вью в браузере (Chrome MCP). Основной способ сейчас.**
   - В таблице есть **отдельная вью «Подписано — ждут возврата»** с фильтром только по `Статус = подписано. положили в папку Бэлле` и сортировкой по `Дата статуса` ↑ (без фильтра по дате — поэтому полная):
     `https://app.notion.com/p/03f3c1ed71b2462ebc69495b83353f4e?v=38e612c762af81c0a71e000cc0b18615`
   - Открыть её и собрать все строки (она уже отфильтрована — клиентский фильтр по статусу не нужен, просто берём всё). Сборщик через `javascript_tool`:
     ```js
     (async () => {
       const sleep = ms => new Promise(r=>setTimeout(r,ms));
       const sc = [...document.querySelectorAll('div')]
         .filter(d => { const s=getComputedStyle(d);
           return (s.overflowY==='auto'||s.overflowY==='scroll') && d.scrollHeight>d.clientHeight+50; })
         .sort((a,b)=>b.scrollHeight-a.scrollHeight)[0] || document.scrollingElement;
       const found = new Map();
       const harvest = () => document.querySelectorAll('.notion-collection-item').forEach(r => {
         const t = r.innerText||''; if (t.trim())
           found.set((t.split('\n')[0]||'').trim(), t.replace(/\n+/g,' | ').trim()); });
       sc.scrollTop=0; await sleep(700); harvest();
       let last=-1;
       for (let i=0;i<50;i++){ sc.scrollTop+=sc.clientHeight*0.85; await sleep(300); harvest();
         if (sc.scrollTop===last) break; last=sc.scrollTop; }
       return { count: found.size, rows: [...found.values()] };
     })();
     ```
   - Колонки в `innerText` этой вью идут в порядке: ФИО · Дата статуса · Сумма возврата · Продукт · Тип · Откуда возврат · Оплата · Статус.
   - Если оператор просит **другой статус** — есть две опции: либо завести/использовать аналогичную вью под нужный статус, либо открыть общую вью таблицы и фильтровать строки клиентски по полю `Статус` (тогда добавить проверку `t.includes('<нужный статус>')` в `harvest`).
   - ⚠️ Не полагаться на старую вью `v=6d98855395194e7da4d418328f960aa0` для полного списка: она отфильтрована по `Дата статуса` после 01.04.2026 и подгружается порциями — легко пропустить строку (проверено 29.06.2026: скролл-сбор по ней дал 3 строки, а чистая вью по статусу — 4).

### Что отдать оператору

Markdown-таблицу + короткий итог. Пример:

> Сейчас в статусе «подписано, в папке у Бэллы» — 3 возврата (дольше всех ждёт верхний):
>
> | ФИО | Статус с | Сумма | Продукт | Тип |
> |---|---|---|---|---|
> | Богданова Мария | 12.06.2026 | 3 600 ₽ (+4 500 на сертификат) | INT02-202605 | частичный |
> | … | | | | |

Это естественная «приёмка» перед боевым `/refund`: видно, по кому пора делать обратные транзакции в Озме.

## Сертификатные возвраты

Отличаются от программных:

- `account_from = 4434` (ООО, Точка) — обычно по платёжке.
- `account_to` — сам **сертификат-аккаунт** в `fin.accounts` (`type = 'Сертификат'`, `contractor` = получатель). Номер из заявления (`№ NNNNN`) ищется как `ext_certificate_number` (он же = `tks_order_id` исходной продажи сертификата).
- `customer` — получатель сертификата (тот, кто подаёт заявление), а не покупатель.
- `name`: `Возврат Фамилия И. О. (сертификат № N)`.
- `linked_transaction` — id исходной продажи сертификата.
- Проверяет, что сертификат-аккаунт уже неактивен (`is_active = false`). Если нет — ставит.
- `request`/`cancellation_status` обычно не трогает (у сертификатной заявки покупателя статус остаётся как был).

## Важно

- **Всегда спрашивать подтверждение перед вставкой**, даже если запрос выглядит однозначным.
- `transaction_date` — это **сегодня** (день внесения в Озму), не дата на PDF. Дата заявления используется только в назначении банковского платежа.
- `tks_state = CONFIRMED` обязательно — иначе транзакция не появится в основном view `fin.transactions_table_all` (там жёсткий фильтр по статусам Т-банка).
- `linked_transaction` обязателен — по нему агрегируются связки оплата↔возврат в PL и отчётах.
- Имя транзакции совпадает с исходной (`Покупатель: <Имя Фамилия> \nПродукт: <CODE>`). Сертификаты — отдельный формат.
- Бонусы: триггер автоматически снимает 5 % (или 100 % для МК) с возвращённой части. Если клиент изначально расплачивался списанными баллами и эти баллы нужно вернуть — добавлять отдельной записью в `base.loyalty_bonuses` с `is_refund_bonuses=true` и положительным `value`, оператор скажет сумму.
- При формате `#NNNN` в ответе — Claude Code UI превратит это в ссылку на GitHub. Писать `транзакция 37574` или markdown-ссылку на view Озмы.
- Не править вручную — править мастер-промпт в Notion, чтобы изменения попали всем.
