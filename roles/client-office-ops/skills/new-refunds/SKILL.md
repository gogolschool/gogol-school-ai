# /new_refunds

Пакетная обработка новых возвратов: проверяем «Анкеты на возврат» с сайта за период, сопоставляем со строками в Notion-таблице возвратов, заполняем форму возврата в Ozma, обновляем статус в Notion.

## Контекст, который нужно подтянуть перед работой

Эти источники меняются — **всегда** читать актуальные перед задачей через `notion-fetch`:

- **Инструкция по форме возврата в CRM (механика)** — `366612c762af81888363cae38b6dc9ad`. Точное описание полей `crm.refund_requests`, какая кнопка какой шаблон, как пишутся реквизиты. Из неё берём, что в какое поле кладём.
- **Алгоритм возврата (бизнес-процесс)** — `29d612c762af80e784d1d10a3b30c669` (актуальная NEW версия) и `18eefb7838c64fa4b39be573a54704d6` (старая, на всякий случай).
- **Notion-таблица «Таблица возвратов»** — `03f3c1ed71b2462ebc69495b83353f4e` (data source `3d37e5a4-902d-4dd7-8f77-55d761fe42ba`).

## Входные данные

Команда принимает диапазон дат в свободной форме:

- `/new_refunds за прошлую неделю`
- `/new_refunds с 13.05.2026`
- `/new_refunds с 01.05.2026 по 15.05.2026`
- `/new_refunds за май`

Парсим в `date_from` / `date_to` (DD.MM.YYYY). Если конец не указан — `date_to = сегодня`.

## Источники данных

| Что | Где | Идентификатор |
|---|---|---|
| Анкеты на возврат с сайта | Bitrix форма «Анкета на возврат» | `form_id = 7`, sid = `ANKETA` |
| Таблица возвратов (контроль статусов) | Notion | data source `3d37e5a4-902d-4dd7-8f77-55d761fe42ba` |
| Заявки студентов | Ozma | `crm.actions_for_contacts` |
| Карточка возврата | Ozma | `crm.refund_requests`, form `crm.refund_view_form` |
| Контакты студентов | Ozma | `base.contacts` / `base.people` |

## Шаги

### Шаг 1 — собрать новые анкеты с сайта и сопоставить с Notion

1. Запросить анкеты:
   ```
   mcp__gogol-school-site__forms_results(
     form_id=7,
     date_from=<DD.MM.YYYY>,
     date_to=<DD.MM.YYYY>,
     include_values=true,
     page_size=500
   )
   ```
   В каждой анкете значимые поля (по `sid`): `SERVISE` (продукт), `SURNAME`/`NAME`/`OTCHESTVO`, `EMAIL`, `PHONE`, `TELEGRAM`, `RS`/`KS`/`BICJAR`/`NAMEOFTHEBANK` (реквизиты), `PURPOSEOFPAYMENT` (для рассрочки), `REASON` (причина), `DATEBIRTH`. Группа `*DOP` — данные получателя если возврат на другого человека.

2. Подтянуть Notion-таблицу:
   ```
   notion-fetch на collection://3d37e5a4-902d-4dd7-8f77-55d761fe42ba
   ```
   Берём строки без терминального статуса (всё что не `деньги возвращены`, не `передумал`, не `перевод в сертификат`).

3. **Сопоставление**: каждой анкете подобрать строку Notion по `Почта` (точное совпадение) → fallback на `ФИО` (нормализация: lowercase, без лишних пробелов). Если матча нет — это новая заявка, **создать строку в Notion** (Шаг 4 ниже).

### Шаг 2 — найти заявку студента в Ozma

Из анкеты у нас: `EMAIL`, `ФИО`, `SERVISE` (название продукта в свободной форме, типа «Лаборатория Маски»). Нужно найти `crm.actions_for_contacts`:

1. По `EMAIL` находим контакт:
   ```
   funql_query: SELECT id, main_name FROM base.contacts WHERE email_sys = '<email>'
   ```
   Если несколько — взять с непустым `email`/`phone`.
   Если ноль — `funql_query` по `main_name LIKE '%<Фамилия>%<Имя>%'`.

2. По контакту находим **активные** заявки:
   ```
   funql_query: SELECT id, action=>name as product, action_datetime, request_status, cancellation_status
   FROM crm.actions_for_contacts
   WHERE contact = <contact_id>
   AND request_status IN (6, 7, 11, 13, 15, 18)  -- обучается / прошёл / оплатил
   AND COALESCE(cancellation_status, '') NOT IN ('Деньги возвращены', 'Перевод в сертификат')
   ORDER BY action_datetime DESC
   ```

3. Из полученных выбрать ту, у которой `action=>name` лучше всего матчится с `SERVISE` из анкеты (нормализуем: оба в lowercase, ищем по ключевым словам — название интенсива/лаборатории/курса, код продукта). Если матч однозначный — берём. Если несколько кандидатов — показать список и спросить.

4. Проверить, нет ли уже `crm.refund_requests` для этой заявки:
   ```
   SELECT id, cancel_type, cancel_request_date, refund_amount FROM crm.refund_requests WHERE request = <request_id>
   ```
   Если есть и заполнено — **не дублировать**, только сообщить пользователю.

### Шаг 3 — собрать сводку и спросить

Показать пользователю таблицу:

| # | Анкета | Дата | ФИО | Продукт | Notion-статус | Ozma заявка | Возврат в Ozma |
|---|---|---|---|---|---|---|---|
| 1 | #123 | 13.05.2026 | Иванова А.О. | LCOURSE01-202604 | написать | id=54982 | нет |
| 2 | #124 | 14.05.2026 | … | … | заполнена анкета на возврат | id=54920 | id=47 (черновик) |

Спросить:
> Внести в Ozma по всем X заявкам?

Если по конкретной строке проблемы (контакта нет, заявок несколько, продукт не сматчили) — выделить и предложить пользователю выбрать/уточнить.

### Шаг 4 — внести данные в Ozma (после подтверждения)

Для каждой строки **создать запись в `crm.refund_requests`** (если ещё нет) через `mcp__OzmaMCP__transaction` с операцией `insert`. Маппинг полей (из инструкции `366612c762af81888363cae38b6dc9ad`):

| Поле в `crm.refund_requests` | Источник | Замечания |
|---|---|---|
| `request` | id заявки из шага 2 | ref |
| `cancel_request_date` | дата отправки анкеты (`createdTime`) | ISO datetime |
| `cancel_type` | из Notion `Тип возврата`: `полный` → `Полный возврат`, `частичный` → `Частичный возврат` | если в Notion пусто — спросить |
| `cancellation_to` | по контексту анкеты: если есть `*DOP` поля → «На карту другого человека», если `PURPOSEOFPAYMENT` непуст → «Через рассрочку Т-Банка», иначе — «На другую свою карту» / «На свою карту (которой платил)» | уточнять у пользователя при неоднозначности |
| `total_service_amount` | посчитать как сумму оплат: `SELECT SUM(CASE WHEN from_our_organization THEN -amount ELSE amount END) FROM fin.transactions WHERE request = <request_id> AND tks_state IN ('CONFIRMED','AUTHORIZED','SIGNED') AND NOT is_deleted` | если есть бонусы — учесть отдельно |
| `refund_commission_pct` | 2.6 (default) | если из Notion `Размер комиссии` другой — переопределить |
| `refund_amount` | **не заполняем** на этом этапе — `auto_refund_amount` посчитается триггером | пользователь потом сверит и при необходимости перебьёт |
| `cancel_account` | если есть бан-счёт у контакта — триггер `auto_fill_refund_request` подставит сам. Если возврат за сертификат — найти `fin.accounts` контакта с `type='Сертификат'` и поставить туда. | |
| `refund_requisites` | склеить из анкеты: `Счет получателя: <RS>\nБИК: <BICJAR>\nБанк: <NAMEOFTHEBANK>\nКорр. счёт: <KS>` | если хотя бы одно поле пустое — пометить и сказать пользователю |
| `order_num` | из `PURPOSEOFPAYMENT` если рассрочка | |
| `cancel_purpose` | `PURPOSEOFPAYMENT` целиком если рассрочка | |
| `refund_reason` | смап `REASON` из анкеты на enum: `Болезнь`, `Травма`, `COVID`, `Нет времени`, `Переезд`, `Не нравится мастер`, `Семейные обстоятельства`, `Работа (командировка)`, `Конфликт`, `Отмена программы`, `Мы отказали`, `Другое` | |
| `refund_reason_commet` | свободный текст из анкеты | |

После вставки — сообщить пользователю:
> ✅ Создал N возвратов в Ozma:
> - [Иванова А. О. — refund_view_form?id=47](https://ozma.gogol.school/views/crm/refund_view_form?id=47)
> - …
>
> Пересчитать сумму возврата для каждого? Сейчас стоит `auto_refund_amount` — он считается с учётом модулей и комиссии. Если нужно перебить вручную — скажи по какому id.

Если возникают вопросы (нет паспорта в Ozma, неоднозначный продукт, странная сумма) — спрашивать по ходу, не блокировать всю пачку.

### Шаг 5 — обновить статус в Notion (после подтверждения)

Спросить:
> Обновляю статус в Notion на «подготовка заявления» и ставлю сегодняшнюю дату?

Если **да** — для каждой обработанной строки Notion:
```
notion-update-page командой update_properties:
  "Статус": "подготовка заявления"
  "date:Дата статуса:start": <ISO today>
```

Если пользователь говорит другой статус (например, «запросили доки», «деньги возвращены», «передумал») — поставить тот.

## Подводные камни

- **Анкета без оплаты на сайте** — иногда студент заполняет анкету, хотя оплачивал не через сайт (наличные / Т-банк рассрочка / др.). В Bitrix-заказах его не будет, но в Ozma `fin.transactions` запись есть. Поэтому источник правды для `request` — Ozma, а не Bitrix-заказы.
- **«Анкета на возврат» = клиент уже хочет вернуть деньги.** Это статус Notion `заполнена анкета на возврат`. После создания записи в Ozma пользователь обычно сразу хочет сменить на `подготовка заявления`.
- **Несколько активных заявок у контакта** — например, человек на двух интенсивах. Если `SERVISE` неоднозначен — спросить.
- **Получатель «другой»** — если в анкете заполнены поля `SURNAMEDOP`/`NAMEDOP`/`OTCHESTVODOP`/`DATEBIRTHDOP`, ставим `cancellation_to = На карту другого человека` и заполняем `cancel_person_last_name`/`first_name`/`patronymic` в `crm.refund_requests`.
- **Сертификатная анкета** — если в SERVISE упоминается «сертификат», ставим `cancel_account` на сертификатный счёт контакта (тип `Сертификат` в `fin.accounts`). Дальше юзер увидит кнопку «Заявление (Сертификат)».
- **Не дублировать!** Перед вставкой всегда проверять `SELECT id FROM crm.refund_requests WHERE request = <X>`. Если уже есть — обновляем существующую, а не создаём новую.

## Какие статусы в Notion есть (для шага 5)

`написать`, `запросили доки`, `заполнена анкета на возврат`, **`подготовка заявления`** (по умолчанию после внесения в Ozma), `заявление на согласовании`, `заявление согласовано`, `позвали/отправили на подпись`, `подписано. положили в папку Бэлле`, `деньги возвращены`, `перевод в сертификат`, `перевод на другого человека`, `передумал`, `возврат скидки`.

## Что НЕ делать

- Не создавать транзакции возврата (`fin.transactions`) — это делает скил `/refund` уже после подписания заявления.
- Не отправлять письма / Telegram — это руками или другим скилом.
- Не менять статус в Notion на терминальный (`деньги возвращены`) — это финальный шаг после фактической выплаты.
- Не править триггерные поля (`auto_refund_amount`, `refund_amount_text`) — посчитаются сами.
