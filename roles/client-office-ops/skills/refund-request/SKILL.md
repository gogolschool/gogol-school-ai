# /refund-request

> ## 🔒 Железное правило: скилл сам НИЧЕГО не отправляет
>
> Единственный исходящий артефакт скилла — **черновик** в папке Drafts ящика (`mcp__yandex-mail__create_draft`). Отправляет письмо всегда человек из веб-почты.
>
> - **Никогда не вызывать `mcp__yandex-mail__send_message`** — ни при каких условиях, даже если когда-нибудь на сервере включат `YANDEX_MAIL_ALLOW_SEND`. Вторая линия защиты: флаг на сервере выключен, и включать его без отдельного решения Бэллы нельзя.
> - **Не отправлять сообщения в Telegram** (студентам, в чаты, Бэлле) в рамках этого скилла — даже уведомления. Если оператор хочет пост в TG-чат возвратов, это отдельная явная просьба вне скилла.
> - Не слать ничего через Unisender, WAZZUP, ботов и любые другие каналы.

Первое звено конвейера возвратов (план: `refund_agent_plan.md` в корне репо):

**/refund-request** (письмо → ответ) → студент заполняет анкету → **/new-refunds** (анкета → `crm.refund_requests`) → заявление/подпись → **/refund** (обратная транзакция).

## Входные формы

- `/refund-request` — разобрать **только новые** письма INBOX: непрочитанные, пришедшие **за последние 2 суток** (или с момента прошлого прогона). Старые непрочитанные (бэклог) НЕ трогать и НЕ вываливать пачкой — упомянуть одной строкой («ещё N старых непрочитанных») и разбирать только по явной команде «разбери бэклог/всю почту».
- `/refund-request за сегодня / с 01.07` — то же по диапазону дат (`list_messages` с `since`).
- `/refund-request uid=1928` — конкретное письмо.
- `/refund-request <пересланный текст письма>` — без почты (оператор обязан указать дату письма — от неё считается правило 7 дней).

## Шаг 1 — чтение и классификация писем

Инструменты `mcp__yandex-mail__*`: `list_messages(unseen_only=true)` → `get_message(uid)` (тело + вложения). Каждое письмо отнести к типу:

| Тип | Признаки | Действие |
|---|---|---|
| **Новый запрос на возврат** | просит вернуть деньги, отменить запись/участие | полный цикл ниже |
| **«Анкета заполнена»** | студент пишет, что заполнил анкету на возврат | проактивная ветка «Анкета заполнена» ниже: карточка возврата создаётся сразу, отчёт фактом со ссылкой |
| **Ответ в существующем кейсе** | реплай в цепочке / в Notion уже есть нетерминальная строка по этой почте | доложить оператору статус кейса, предложить следующий шаг конвейера (анкета → /new-refunds; скан заявления → сохранить и передать) |
| **Выбор «сертификат»** | отвечает согласием на перевод в сертификат | доложить; статус отмены «перевод в сертификат» — по подтверждению |
| **Прислано подписанное заявление** | вложение-скан, «заявление» в теме | скачать вложение (`download_attachment`), доложить оператору — это этап /refund, не наш |
| **Возврат подаренного сертификата** | пишет получатель серта (часто не покупатель и с чужой почты), номер/PDF сертификата | найти серт в `fin.accounts` по `ext_certificate_number`; проверить срок действия (год с оформления): **истёк → эскалация** (по правилам деньги после срока не возвращаются, решает Бэлла); действует → сертификатный сценарий: анкета + заявление (Сертификат), детали в /refund |
| **Запрос статуса / пинг** («когда ответите?», «уточните статус») | вопрос без нового запроса | поднять кейс (Notion + СРМ + переписка), доложить в чат с датами («ждёт ответа с 29.06 ⚠️»), подготовить черновик ответа студенту |
| **Просит перенос вместо возврата** (на другой поток/продукт) | «переведите меня на…» | это НЕ возврат: статус отмены «Перевод в другую лабу/интенсив» (id 14) или «Перевод на следующий поток» (id 22), доложить оператору — перенос оформляют админы |
| **Ошибочное списание / проблема оплаты** (списалось с карты вместо сертификата, двойная оплата) | описывает ошибку, не отказ от услуги | не конвейер писем: эскалация оператору, исправление — возврат по транзакции через /refund |
| **Не про возврат** | всё остальное | перечислить в отчёте, не трогать |

Письма **не помечать прочитанными** до конца обработки (`mark_read` — только после подтверждённых действий по письму).

## Шаг 2 — идентификация студента в Ozma

1. Контакт по адресу отправителя — **через `base.communication_ways`** (`email_sys` у многих пуст, проверено 06.07.2026):
   ```
   SELECT cw.contact, cw.contact=>main_name AS name FROM base.communication_ways AS cw
   WHERE cw.type = 'Email' AND LOWER(cw.data) = '<email в lowercase>' AND NOT cw.is_deleted
   ```
   Пусто → fallback `WHERE LOWER(email_sys) = …`, затем поиск по ФИО из подписи. Несколько → показать варианты.
2. Заявки контакта (поле даты — `create_datetime`; `cancellation_status` — reference на `crm.cancellation_statuses`, сравнивать по `=>name`):
   ```
   SELECT id, action, action=>name AS product, create_datetime,
          request_status=>name AS status, cancellation_status=>name AS cancel_st
   FROM crm.actions_for_contacts
   WHERE contact = <id>
   ORDER BY create_datetime DESC
   ```
   Кандидаты на возврат — заявки со статусом «Обучается»/«Оплатил» и пустым/нетерминальным статусом отмены; заявка со статусом «Отмена» тоже валидна, если продукт отменяли мы (кейс «Отмена программы»).
3. Продукт — матчинг названия из письма с `action=>name` (ключевые слова, код). Неоднозначно → спросить оператора. Письмо может быть от родственника/с другой почты — если контакт не находится, не угадывать, спросить.
4. Оплаты по заявке:
   ```
   SELECT id, amount, transaction_date, account_to, account_to=>name AS acc_name, tks_state
   FROM fin.transactions
   WHERE request = <request_id> AND tks_state IN ('CONFIRMED','AUTHORIZED','SIGNED') AND NOT is_deleted
   ```
   Маппинг счёта → поля Notion «Оплата»/«Откуда возврат» (сверять по `acc_name`, id могут меняться): 6 = Т-Банк эквайринг, 26728 = CloudPayments, 23719 = Яндекс Сплит, 4434 = ООО (Точка); рассрочки/Долями/Сплит видны по названию счёта.
5. Дата старта программы — первое занятие продукта. Занятия = **дочерние строки `crm.actions`** (`parent_action` = id продукта, `type = 'Занятие'`):
   ```
   SELECT MIN(action_datetime) AS first_lesson, MAX(action_datetime) AS last_lesson, COUNT(*) AS cnt
   FROM crm.actions WHERE parent_action = <action_id> AND NOT is_deleted
   ```
   ⚠️ `action_datetime` в UTC — для правила «7 дней» сравнивать даты по МСК (+3).
6. Бонусы: `loyalty.loyalty_card.calculated_current_bonus` + оплаты баллами по заявке (если платил баллами — учесть в расчёте и в письме).

## Шаг 3 — дедупликация

- Notion: строка с этой почтой и **нетерминальным** статусом (терминальные: `деньги возвращены`, `передумал`, `перевод в сертификат`, `перевод на другого человека`, `возврат скидки`) → это существующий кейс, новый не заводить.
- Ozma: `SELECT id FROM crm.refund_requests WHERE request = <request_id>` → уже есть карточка → доложить, не дублировать.
- `cancellation_status` заявки уже «Попросил возврат» → статусы не перетирать.

## Шаг 4 — расчёт правила и сумм (по оферте)

Вход: дата письма (= дата запроса), дата старта, оплачено всего, способ оплаты, посещённые занятия (если после старта), бонусы.

| Ситуация | Формула возврата | «Размер комиссии» в Notion |
|---|---|---|
| Запрос **> 7 календарных дней** до старта | оплачено − комиссия банка **2.6%** | `2.6%` |
| Запрос **≤ 7 дней** до старта | оплачено − комиссия 2.6% − **7%** от стоимости услуг (оферта п. 5.9) | `7% + 2.6% (менее 7 дней до начала)` |
| **После начала** занятий | оплачено × (1 − удержание); удержание по модулям 30/50/20, **начатый модуль — пропорционально прошедшим занятиям внутри него** (формула ниже). **Комиссия 2.6% НЕ удерживается** — п. 5.10 оферты её не предусматривает (комиссия только в сценариях до начала, п. 5.9) | частичный |
| Рассрочка Т-Банк / рассрочка CloudPayments | те же правила по срокам/модулям, но **комиссия 2.6% НЕ удерживается**; возврат уходит на счёт рассрочки, у студента просим только паспорт | + блок «Рассрочка» в письме |
| Долями / Яндекс Сплит | те же правила, комиссия 2.6% **удерживается**; возврат через сервис | + блок «Долями»/«Сплит» в письме |
| Оплата (частично) сертификатом / баллами | деньги и сертификатную часть считать раздельно; баллы — упомянуть аннулирование | |

**Источник правды правил расчёта — Google-таблица «Суммы возвратов»** (`1eRtJVz9xugFTxKuOuoSVet4Xmvxjx25u0gQd16geE_s`, листы «модули с 1.03.2026» и «формулы для расчета возврата и бонусов»; перед расчётом сверяться — правила могут обновить). Правила действуют **для продуктов с 1.03.2026** (для более старых — лист «старое»).

Модульная сетка (доли всегда 30% / 50% / 20%, отличается размер модулей):
- **Лаборатория (24 занятия)**: М1 = первые 4 занятия, М2 = следующие 4, М3 = остальные 16.
- **Курсы и интенсивы > 8 занятий, HEAD, ППВ**: М1 = 2 занятия, М2 = 2 занятия, М3 = остальные.
- **Интенсивы/курсы ≤ 8 занятий**: М1 = 1 занятие, М2 = 2 занятия, М3 = остальные.

**Формула удержания (критично: начатый модуль считается ПРОПОРЦИОНАЛЬНО, не целиком)** — как в листе «формулы для расчета возврата и бонусов» и в триггере Озмы `calc_auto_refund_amount`:

`удержание = min(прошло, М1)/М1 × 30% + max(0, min(прошло, М1+М2) − М1)/М2 × 50% + max(0, прошло − М1 − М2)/М3 × 20%`

Пример (лаба 24 зан., прошло 5, оплачено 81 000): 4/4 × 30% + 1/4 × 50% = **42.5% удерживаем** → возврат = 81 000 × 57.5% = 46 575 ₽, без комиссии; «в сертификат» после начала — та же сумма. Свой расчёт сверять с `auto_refund_amount` в карточке — триггер считает по этой же формуле; расхождение = флаг оператору.

Перевод в сертификат:
- **До начала программы** — полный номинал (без комиссии и удержаний).
- **После начала** — по модулям (та же сетка, что и возврат).
- **После начала со справкой** (болезнь/командировка): ни разу не был ИЛИ начал ходить + справка — **по занятиям** (пропорционально фактически прошедшим занятиям, а не модулям).

Обязательные проверки:
- Сумма «при переводе в сертификат» — по правилам выше. Сложные кейсы со справками — сумму подтверждает оператор.
- **Комиссия банка 2.6%** применяется ТОЛЬКО в сценариях до начала занятий (п. 5.9: >7 дней и ≤7 дней) и только при оплате эквайрингом (Т-Банк, CloudPayments), Долями или Сплитом. НЕ удерживается: при рассрочках (Т-Банк, CloudPayments), в частичных возвратах после начала (п. 5.10) и при переводе в сертификат. Ярлык «2.2%» в таблице — легаси для старых строк, в новых кейсах не использовать.
- Отмена программы **с нашей стороны** — возврат полный без удержаний, письмо другое (уточнить у оператора).
- **ОУ / пробные занятия** — правило подтверждено Бэллой 07.07.2026: по оферте пробного (https://gogolschool.ru/docs/dogovor-ou/) удержания 7% **нет** — возврат минус только банковская комиссия 2.6%, независимо от срока до занятия. Возвраты ОУ обрабатывать как обычные кейсы: строка в таблице возвратов («Возврат за: ОУ»), статусы, черновик ответа.

## Шаг 5 — сводка и подтверждение (ничего не пишем до «да»)

Формат согласован Бэллой 07.07.2026. **Одно сообщение на одно письмо** (не пачкой), в каждом — тег Светы **@Beverkar2**. Механика в Telegram-боте: блоки, разделённые строкой `---`, отправляются отдельными сообщениями — каждый кейс отделяй `---`, ничего не объединяй в общую канву. Суммы всегда парой: «При возврате: …» / «При переводе в сертификат: …» (обе цифры уйдут и в письмо студенту). Заканчивать одним вопросом.

Стандартный кейс:

> ✉️ **Новый запрос на возврат** @Beverkar2
>
> **Иванова Мария** — письмо сегодня в 14:20
> Программа: **«Спонтанность и игра»** (INT02-202607)
> Оплачено: **17 500 ₽**, CloudPayments эквайринг, 03.07
> Старт: 14.07 → запрос **за 8 дней** = удержание только комиссии 2.6%
>
> При возврате: **17 045 ₽**
> При переводе в сертификат: **17 500 ₽**
>
> Проверила: в таблице возвратов человека нет, карточки возврата нет, повторных писем нет.
>
> Мои шаги дальше:
> 1. Поставлю в заявке статус **«Отмена»**, статус отмены — **«Попросил возврат»**
> 2. Создам запись в таблице возвратов в Notion и пришлю ссылку
> 3. Подготовлю черновик ответа с суммами и анкетой — покажу текст перед сохранением
>
> **Делаю? (да / нет / обсудить)**

**Нестандартные кейсы** — тот же формат, но заголовок «✉️ Новый запрос на возврат — ⚠️ нужно решение», явный флаг, что именно нестандартно, и сумму сама не утверждать. Список нестандартных:
- **Отмена программы школой**: не состоялась вовсе → возврат полный, без удержаний. Завершена досрочно **после начала** → возврат **по фактически проведённым занятиям** (не по модулям; прецедент 07.2026: 3 из 8 занятий → вернули 5/8 суммы), потраченные бонусы вернуть; школа может добавить подарочные баллы (решает оператор). Также предложить перенос на следующий поток без доплат.
- **Реквизиты прислали письмом** (без анкеты) → анкета всё равно обязательна (реквизиты из письма не использовать) — в черновике ответа вежливо направить на анкету.
- **Мы не пустили / отказали в участии** (например, несовершеннолетний, не подходит по правилам) → возврат полный без удержаний; в CRM статус отмены «Мы отказали» (id 2), причина «Мы отказали».
- **Серьёзная причина: болезнь, командировка** (и другое, что подтверждается справкой) → возвращаем добрее: попросить справку; по шаблону «Частичный возврат при болезни/командировке» (uid 14 в папке «Шаблоны») предложить выбор — денежный возврат по модулям ИЛИ **сертификат по расчёту ПО ЗАНЯТИЯМ** (выгоднее для студента). В письме два вопроса: есть ли справка и какой вариант выбирает. В сводке показать три суммы: возврат по модулям / сертификат по модулям / сертификат по занятиям.
- **Спор о сумме / несогласие с модульной системой** → эскалация Бэлле/Кате, ничего не считать финально.
- **Платил другой человек / контакт не найден / несколько подходящих заявок** → уточнить у оператора.
- **Повторный запрос по уже завершённому возврату** → жирный флаг, ничего не делать без решения.

Другие типы писем — коротко, одной строкой, тоже с тегом: реплай в кейсе («**Лупенских** ответила в цепочке по INT02 — выбрала сертификат, переключаю статус?»), скан заявления («**Шинкаренко** прислала скан заявления 📎 — скачала, передаю»), не про возврат — упомянуть в конце разбора без отдельного сообщения.

## Ветка: письмо «анкета заполнена»

Когда студент пишет, что заполнил анкету на возврат, — это **проактивная ветка** (решение Бэллы 07.07.2026): карточку возврата Лана создаёт сразу, без предварительного «да», и отчитывается фактом. Карточка — внутренний документ CRM, клиенту ничего не уходит.

1. **Найти анкету на сайте**: `mcp__gogol-school-site__forms_results` (form_id=7, ANKETA) за последние дни, матч по email. Нет анкеты → сообщить в чат («пишет, что заполнил, но анкеты на сайте нет — попросить заполнить ещё раз?»), дальше не идти.
2. **Статусы в СРМ**: если у заявки ещё не стоят «Отмена» (9) + «Попросил возврат» (4) — проставить (и отметить это в отчёте).
3. **Создать карточку возврата** `crm.refund_requests` по маппингу из /new-refunds (реквизиты, причина, суммы, `total_service_amount`). Важное:
   - `cancel_request_date` = дата **первичного письма-запроса** (из строки Notion «Дата запроса»), не дата анкеты; если первичной даты нет — дата анкеты.
   - Сломанный триггер `calc_late_cancel_deduction`: insert делать **без** `cancel_request_date`, затем update с `raw:true`.
   - `cancellation_to` заполнить по эвристике (есть `*DOP`-поля → «На карту другого человека»; рассрочка → «Через рассрочку Т-Банка»; иначе «На свою карту (которой платил)») и **обязательно пометить в отчёте**, что поле нужно проверить — из анкеты оно технически не приходит.
   - Перед созданием — проверка дубля: `SELECT id FROM crm.refund_requests WHERE request = <id>`; есть → обновить существующую.
4. **Notion**: статус строки → «подготовка заявления», дата статуса = сегодня.
5. **Отчёт в чат** (формат Бэллы):
