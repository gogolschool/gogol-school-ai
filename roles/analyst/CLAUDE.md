# Роль: analyst (аналитик)

> Общий контекст про Gogol School — в [shared/CLAUDE.md](../../shared/CLAUDE.md). Этот файл — про специфику аналитика. Будет расширяться по мере накопления практики анализа.

## Зона ответственности

Аналитика по бизнесу Gogol School:

- **Студенты**: воронка, активные / спящие, повторные покупки, churn
- **Продукты**: выручка, маржинальность, наполняемость, эффективность форматов
- **LTV / Retention**: метрики жизненного цикла клиента и удержания
- **Когортный анализ**: разрезы по месяцу первой покупки, формату продукта
- **Маркетинг**: эффективность каналов, конверсии, ROI кампаний

## Источники данных (порядок проверки)

## Правила подсчёта — обязательное чтение перед задачей

### Доход, возвраты, выручка

Полные формулы — в [shared/knowledge/database.md](../../shared/knowledge/database.md#-правила-подсчёта-дохода-возвратов-и-выручки-критично) и в памяти ассистента.

Кратко:
- **Доход** = `tks_state IN ('CONFIRMED','AUTHORIZED')` AND `from_our_organization = false`
- **Возвраты** = `tks_state IN ('CONFIRMED','REFUNDED','PARTIAL_REFUNDED')` AND `from_our_organization = true`
- **Выручка** = доход − возвраты
- Статус `Ожидается оплата` — **никогда не учитывается**

### LTV / Retention — единица покупки

📐 **Полная страница**: `33b612c762af812aa1bdfde63d15bb26` («Правила подсчёта покупок для аналитики LTV, Retention»). Подтягивать через `notion-fetch` перед каждой задачей по этим метрикам — правила могут обновиться.

#### Общий принцип

**1 заявка = 1 покупка** (`crm.actions_for_contacts`), независимо от числа транзакций внутри. Для абонементов: **1 абонемент = 1 покупка** (`subscription.subscription_info`).

#### Блок A — обычные заявки

- Источник: `crm.actions_for_contacts`
- Сумма покупки = сумма транзакций (`fin.transactions.request`), при условии:
  - `is_deleted = false`
  - `is_plan = false`
  - транзакция **не** является оплатой абонемента (нет ссылки из `subscription_info.payment`)
  - `tks_state != 'Ожидается оплата'`
- **Исключаем заявку**, если одновременно: статус «Отмена» **И** статус отмены «Возврат полный произведён».
  - Частичный возврат, перевод в сертификат, перенос — **не исключаются**, остаются в аналитике.
- Если **все** транзакции заявки — оплаты абонементов, заявка в блок A **не попадает** (считается в блоке B).
- Смешанных заявок (часть обычные + часть абонементные) не существует.

#### Блок B — абонементы

- Источник: `subscription.subscription_info`
- Сумма покупки = `subscription_info.payment.amount`
- Абонементы бывают:
  - Привязаны к заявке через `subscription.requests_for_subscriptions`
  - Без заявки — **тоже считаются** как отдельная покупка
- **Исключаем** абонементы со статусом `возврат`

#### Временной срез

Учитываем покупки начиная с даты, указанной пользователем (пример: с 2022 = `>= 2022-01-01`). Фильтр — по **дате покупки** (дата заявки / дата создания абонемента), **не** по дате клиента. Клиент попадает в аналитику только с теми покупками, которые в диапазоне.

#### Сводка случаев

| Ситуация | Результат |
|---|---|
| Заявка с 2 обычными транзакциями | 1 покупка в блоке A, сумма = обе транзакции |
| Заявка с транзакциями-оплатами абонементов | В блоке A не считается. Каждый абонемент = отдельная покупка в B |
| Абонемент без заявки | 1 покупка в блоке B |
| Заявка «Отмена» + полный возврат | Не считается нигде |
| Транзакция без заявки с `tks_state = 'Ожидается оплата'` | Не считается (это 106M из 108M таких транзакций) |
| Абонемент со статусом «возврат» | Не считается |
| Частичный возврат по заявке | Считается (остаётся в блоке A) |

#### Источники данных (быстрая навигация)

| Что | Таблица |
|---|---|
| Заявки | `crm.actions_for_contacts` |
| Статусы заявок | `crm.request_statuses` |
| Статусы отмены | `crm.cancellation_statuses` |
| Транзакции | `fin.transactions` |
| Абонементы | `subscription.subscription_info` |
| Связь заявка ↔ абонемент | `subscription.requests_for_subscriptions` |
| Контакты | `base.contacts` |

### Исключения из выручки (cohort-анализ)

Из выручки исключаются: `class_type NOT IN ('HEAD', 'ОУ HEAD', 'B2B', 'Сертификат')`. См. также [shared/knowledge/systems.md §7.3](../../shared/knowledge/systems.md).

## Инструменты в OzmaMCP, которыми пользуюсь чаще всего

- **`funql_query`** — основной инструмент для SQL/FunQL запросов
- **`named_view_query`** — выполнение готовых user views (быстрее и стабильнее, чем сырые запросы)
- **`validate_funql`** — проверка запроса до выполнения (для сложных)
- **`funql_guide`** — справка по синтаксису FunQL
- **`list_entities`, `list_entity_fields`** — навигация по схеме
- **`search_field`, `where_used_field`** — поиск конкретного поля
