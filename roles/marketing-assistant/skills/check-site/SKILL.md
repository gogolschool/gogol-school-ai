# /check-site

## Что делает

Ежедневная проверка **витрины сайта** gogolschool.ru глазами клиента: раздел «Программы» (`/programms/`), карточки программ и блок «ближайшие» на главной. Открывает живой сайт в браузере (Chrome MCP) и добивает данными из Bitrix (iblock 16), формирует Markdown-отчёт о найденных проблемах. Только читает; ничего не правит.

Это **не** сверка данных между системами (Notion ↔ Озма ↔ сайт ↔ календарь) — для этого `/check-products`. Здесь — как сайт реально выглядит и работает для покупателя.

## Как работает

1. **Загрузи актуальный мастер-промпт из Notion:** page `388612c762af8113a1f7f9e799324982`
   (https://www.notion.so/388612c762af8113a1f7f9e799324982). Действуй строго по нему — там карта сайта, что и как проверять (фильтры, порядок, открываемость, ссылки оплаты/sold-out, поля карточки, вёрстка, мастера), что считать проблемой, формат отчёта и правила.
   > Notion берём через **рабочий** MCP (`c2755fd9-…`), не личный (см. память `feedback_notion_workspace_routing`).
2. Открой каталог `/programms/` в Chrome MCP, собери карточки и пройди блоки A–D из промта.
3. Данные (цена, `active`, ссылка оплаты, `service_id`) добирай из iblock 16 через `gogol-school-site`.
4. Сформируй отчёт о проблемах; правки не делай — вынеси как задачи профильным скиллам.

## Критические правила

- **Только отчёт.** Скилл не правит сайт. Никаких `*_update`/`*_create`/`*_deactivate` — сайт продакшн, режим read-only.
- **Браузер бережно.** Не отправляй формы оплаты и не нажимай «купить» до конца — проверяй наличие и адрес ссылки, не инициируй платёж.
- **Каталог/карточки рендерятся JS** — статический `curl` для карточек бесполезен, используй Chrome MCP; данные бери из iblock 16.
- **Делегирование фиксов:** данные продукта/ссылка оплаты/расписание → `/newproduct`; строка в таблице → `/add-product`; расхождение данных между системами → `/check-products`; контент мастеров/вёрстка → задача владельцу сайта.
