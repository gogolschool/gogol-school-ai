# /new-campaign

## Что делает

Заводит в Озме **почтовую маркетинговую** кампанию по ТЗ: создаёт `marketing.campaigns` в статусе **Черновик**, привязывает уже готовые списки (целевые + исключения) через `marketing.lists_in_campaigns`, создаёт задачу распространения в `pm.actions`. На отправку НЕ нажимает — доводит до черновика и передаёт человеку.

Скилл — **ассемблер**. Он НЕ подбирает сегмент, НЕ создаёт/наполняет списки, НЕ клонирует шаблон Unisender, НЕ генерит тему. Всё это — вход от других скиллов/процессов (`template_id`, id списков, тема письма приходят в ТЗ готовыми).

## Как работает

1. **Загрузи актуальный мастер-промпт из Notion:** page `367612c762af8134bc68e3042613a9a7`
   (https://www.notion.so/367612c762af8134bc68e3042613a9a7). Действуй строго по нему — там формат ТЗ, маппинг полей `marketing.campaigns` / `lists_in_campaigns` / `pm.actions`, правила именования кампании и темы, проверка на дубликат.
   > Notion берём через **рабочий** MCP (`c2755fd9-…`), не личный (см. память `feedback_notion_workspace_routing`).
2. **Прими ТЗ** по шаблону из промта (Шаг 1): дата+время отправки (МСК), тема письма без префикса, ID шаблона Unisender, сегмент/суть (для имени), целевые списки, исключения, корзинка. Не хватает обязательного — переспроси, не начинай.
3. **Валидация** (Шаг 2): каждый список найди в `marketing.lists`, корзинку (если не null) — в `crm.baskets`. Не найдено / несколько совпадений — стоп, переспроси.
4. **Дубликат** (Шаг 3): нет ли уже кампании с тем же `name` (или «дата+`template_id`»). Нашёл — покажи и спроси, по умолчанию не создаём.
5. **Покажи сводку и дождись подтверждения** (имя кампании, тема, `template_id`, списки целевые/исключения + размер, корзинка, дата/время). Только после «да» пиши.
6. **Запиши через `mcp__ozma__transaction`** (один транзакцией): `marketing.campaigns` → `marketing.lists_in_campaigns` (по строке на список) → `pm.actions` (задача распространения).
7. **Отчитайся** id кампании, ссылкой на форму (`https://ozma.gogol.school/views/marketing/campaign_form?id=<id>`), составом списков. Напомни, что **отправку** делает человек отдельно.

## Критические правила (выверено по живой Озме 2026-06-30 — в мастер-промте этого нет или иначе)

### 1. `responsible_person` = id 21186, НЕ «текущий пользователь»
Промт пишет «текущий пользователь» (для интерактивного запуска человеком это верно). Но при запуске скиллом/ботом ставь **`responsible_person` = `base.people` id 21186** — «Степанова Ксения для рассылок» (`k.stepanova@gogol.school`), служебный владелец AI-кампаний (см. память `reference_campaign_responsible_person`). Пользователя `it@gogol.school` в базе нет.

### 2. Нет поля `notification_who` — не добавляй
В `marketing.campaigns` и `pm.actions` поля `notification_who` **нет** (оно только в схеме `fin`). Строку `notification_who="Клодик"` из промта удалили 2026-06-30 — если вдруг увидишь её снова, игнорируй. Атрибуция AI-записи обеспечена `responsible_person` (id 21186) + системными `created_person`/`created_datetime`.

### 3. Id справочников (подтверждены по базе)
- `marketing.campaigns.stage` → **Черновик = 4** (Отменено=1, Завершено=2, В работе=3, Ошибка=5).
- `pm.actions.type` → **Рассылка = 3** (канал email фиксирован).
- `pm.actions.stage` → **Новое = 5**.

### 4. Скилл НЕ отправляет
Доводит ровно до **Черновик + задача распространения «Новое»**. «Распространить действие» и «отправить рассылку» — деструктивные внешние шаги, делает человек отдельно после проверки визуала шаблона в Unisender. Авто-отправки по триггеру нет.

### 5. Подтверждение перед записью
Любая запись в прод-CRM — только после показанной сводки и явного «да» (как в `feedback_ozma_refunds`). При аномальном размере списка (слишком большой/малый) предупреди.

### 6. Только email. Бот-каналы — вне области
Промт и скилл — для канала `mail`. Для бот-рассылок `pm.actions.type` другой: **8** = Рассылка в боте абонементов, **9** = Рассылка в боте GS, **10** = WhatsApp. Это отдельный промт/скилл — здесь не делаем.

### 7. МСК без конвертации
Дата+время отправки пишутся в `pm.actions.start_date` как есть, в МСК (промт: «работаем в МСК и пишем в БД МСК»).
