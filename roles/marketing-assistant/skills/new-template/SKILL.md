# /new-template

## Что делает

Клонирует образцовый шаблон письма в Unisender и подставляет новый контент → отдаёт `template_id` для кампании. Образец остаётся нетронутым (создаётся **новый** шаблон). Сам не отправляет — визуал перед отправкой проверяет человек.

## Как работает

1. **Загрузи каталог из Notion** (рабочий MCP `c2755fd9-…`): page `365612c762af809ab78ac99900af1d6b` («🪐 Шаблоны для рассылок»). Там: название → **номер шаблона** (id), список **переменных** каждого шаблона и **общие правила вёрстки**.
2. **Разбери вход:**
   - **Номер (id)** — сразу бери образец, шаг поиска/уточнения пропусти.
   - **Название** (напр. «mktg02 Информация о продукте», «Отмена программы») — найди подходящий шаблон в каталоге, **уточни у пользователя** (тот ли это шаблон: покажи его текущую тему + назначение), и только потом клонируй.
   - **«Уникальный шаблон»** (так выбрано в поле «Шаблон рассылки» карточки) — **клонировать НЕ нужно** и в Unisender ничего не создаём. Готовый нестандартный шаблон уже сделал маркетолог сам — задача только записать его номер в карточку (см. блок «Уникальный шаблон» ниже). Дальше сразу к `/new-campaign`.
3. **Возьми тело образца:** `mcp__unisender__get_template(template_id=<номер>)` — получишь `subject` + полный HTML `body`.
4. **Собери новый контент** под переменные этого шаблона (из ТЗ/карточки; чего нет — спроси). Список переменных — в каталоге у каждого шаблона.
5. **Замени только переменные блоки**, сохранив каркас образца (шапка, футер, вёрстка). Собери новый `body` + `subject`.
6. **Создай новый шаблон:** `mcp__unisender__create_email_template(title, subject, body)` → вернёт `{result:{template_id}}`. Имя осмысленное: `<код образца> — <кампания/продукт> <ДД.ММ>`.
7. **Отчитайся** новым `template_id` и напомни: визуал проверяет человек в Unisender перед отправкой; `template_id` идёт в `/new-campaign`.

## Критические правила

1. **Копия, не правка образца.** Всегда `create_email_template` (новый шаблон). НИКОГДА `update_email_template` по номеру образца — иначе сломаешь эталон для всех.
2. **Сохрани общие правила вёрстки** (из каталога): размеры текста **16 / заголовки 18 / футер 14** (других нет); обязательная **шапка с лого + меню** (программы/мастера/о школе/контакты); обязательный **футер с согласием + отпиской**; цвета акцентов розовый `#fbedea`, красный `#d64f34`, блок `#ededef`; редакторские правила — кавычки-ёлочки « », без буквы «ё».
3. **Ссылки с UTM.** Все ссылки (особенно CTA/кнопка) должны содержать UTM-метки. UTM берётся из карточки/UTM-промта кампании — не оставляй «голую» ссылку.
4. **Меняем только контент переменных** — остальной HTML образца не трогаем. Не путать «переменные» каталога (контентные блоки, которые редактируем) с полями подстановки Unisender (мерж-поля `{{...}}` в письме).
5. **Уточнение при названии.** Если вход — название, перед клоном подтверди с пользователем, что выбран верный образец (покажи назначение + текущую тему). При id — не уточняй, клонируй.
6. **Выход — `template_id`.** Это вход для `/new-campaign` (поле `template_id` кампании). Визуальную проверку и отправку делает человек.
7. **Работай сам(а), без суб-агентов.** НЕ запускай вложенных агентов (Task/Agent) для сборки шаблона — они не укладываются в лимит ответа и обрываются по таймауту. Все шаги делай сам(а) по порядку.
8. **Тело шаблона держи в памяти, не «пили» файл.** Бери `body` прямо из ответа `get_template` и меняй в нём только блоки-переменные обычной строковой заменой. НЕ сохраняй HTML на диск, чтобы потом грепать/резать его через jq/grep — это медленно и лишнее. Один раз получил тело → заменил блоки → `create_email_template`.
