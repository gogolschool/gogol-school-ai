# /add-promocode

## Что делает

Заводит промокод в двух системах согласованно:
1. **Сайт (Bitrix, iblock 17)** — создаёт код в нужной папке-разделе (`promo_create`), со скидкой %/₽, сроком, лимитом.
2. **OzmaDB (`fin.promocodes`, вью `fin/promocodes_table`)** — вносит тот же код с `discount_type`, `active_until`, `max_uses_per_user`, `product/basket`.
3. **Охват «на все программы» через папку** — если код должен работать на всех программах через раздел (напр. 64 «Партнёрские»), проверяет продаваемые услуги и добавляет раздел в `PROMO_CODE_SECTION` тем, где его нет; отдельно отмечает услуги с запретом промокода и абонементы.

## Как работает

1. **Загрузи актуальный мастер-промт из Notion:** page `395612c762af81d6b7c7dc26c685abda`
   (https://www.notion.so/395612c762af81d6b7c7dc26c685abda). Действуй строго по нему — там разделы-папки, поля Bitrix и Ozma, механика охвата через папку и правила.
   > Notion берём через **рабочий** MCP (`c2755fd9-…`), не личный (см. память `feedback_notion_workspace_routing`).
2. Разбери ТЗ: скидка (% или ₽), срок (до какой даты), повод/аудитория, папка-раздел, охват (все программы / конкретный продукт), лимит на пользователя. Чего нет — придумай (код) или спроси (папка, охват).
3. Проверь уникальность кода: сайт `promo_search`/`GET /promo?code=`, Ozma `SELECT ... FROM fin.promocodes WHERE code=`.
4. Создай на сайте (Bitrix) → внеси в Ozma → при охвате через папку добей раздел услугам.
5. Самопроверка из промта, отчёт: ID на сайте, ID в Ozma, сводка охвата.

## Критические правила (набиты практикой в этой ветке)

### 1. Две системы — держать в синхроне
Код заводится и на сайте (Bitrix, iblock 17), и в Ozma (`fin.promocodes`) — это **разные записи**, но `code`/`discount`/срок/охват должны совпадать. Забыть одну систему = промокод «есть», но где-то не работает или не считается.

### 2. «Папка» = свойство услуги, а не глобальный флаг
Промокод применяется к услуге (iblock 16), только если его **раздел** есть в `PROMO_CODE_SECTION` услуги (в API — `promoSections`) **или** код индивидуально в `PROMO_CODE`. Глобального «на всё» флага нет: даже общий код реально привязан к спискам услуг/разделов. Поэтому «на все программы» = добить нужный раздел всем продаваемым услугам.

### 3. read-modify-write, не затирать
`PROMO_CODE_SECTION` (и `PROMO_CODE`) перезаписываются **целиком**. Всегда: `services_get` → взять текущие `promoSections` → записать `[...старые, новый]`. Иначе сотрёшь чужие привязки.

### 4. denyPromocode блокирует всё
У услуги с `denyPromocode=true` («Запретить промокод») код не сработает, даже если папку добавить. Такие услуги выносить отдельным списком и спрашивать, снимать ли флаг (сам не снимай).

### 5. Что считать «продаваемой программой»
Поля `format`/`dostuponost` у услуг почти везде пустые — не опираться. Продаваемые = `active=Y` + `available="1"` + (`activeTo` пусто или в будущем). Из них убрать шум: **Сертификаты**, мерч (наклейки/брелоки/футболки/шопперы/носки/«Мерч…»), **тесты**, **ОУ/пробные** (`OU…`), услуги с `denyPromocode`. Абонементы (в названии «абонемент») — отдельная группа, трогать по явному «и на абонементы».

### 6. Корпоративный код → лимит 1 на пользователя
Для корпоративных/партнёрских кодов в Ozma `max_uses_per_user=1` (каждый сотрудник по разу) — как `DTMSXGS`, `GSXHAWAII`. `product/basket=null` = на все программы.

### 7. Подтверждение перед записью в Ozma и массовой привязкой
Перед insert в `fin.promocodes` и перед пакетным добавлением раздела к услугам — сводка и подтверждение (память `feedback_ozma_refunds`).
