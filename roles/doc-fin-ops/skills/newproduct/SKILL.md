# /newproduct

## Что делает

Переносит карточку продукта из таблицы Ноушн в OzmaDB (`crm.actions`): создаёт продукт, привязывает мастеров (`crm.masters_for_actions`), занятия, проставляет `price` / `split_parts`, обновляет поле «Ссылка в CRM» в карточке Ноушн. Поддерживает интенсивы, ISEEYOU, курсы, маркетинговые МК, открытые уроки (ОУ) и массовое создание за месяц.

## Как работает

1. **Загрузи актуальный мастер-промпт из Notion:** page `31f612c762af81ba999cc5657404528c`
   (https://www.notion.so/31f612c762af81ba999cc5657404528c). Действуй строго по нему — там маппинг полей, фиксированные справочники по типам, нумерация (`INT/ISEEYOU/OU/MK`), формат расписания, конвертация времени и самопроверка.
   > Notion берём через **рабочий** MCP (`c2755fd9-…`), не личный (см. память `feedback_notion_workspace_routing`).
2. Открой карточку(и) продукта по ссылке от пользователя, извлеки все поля.
3. Найди справочные ID в Озме (мастера, локация) — см. «Поиск ID» ниже.
4. **Покажи пользователю сводку** перед вставкой (имя/CODE, тип, даты, расписание, мастер, локация, цена, справочники) и дождись подтверждения — особенно при массовом создании.
5. Создай в Озме, обнови «Ссылка в CRM» в Ноушн, выполни самопроверку из промта, отчитайся ID и ссылками.

## Мастер-промты (режимы)

Сначала определи, что именно просят, и подтяни **нужный** мастер-промт из Notion (рабочий MCP). Действуй строго по выбранному.

| Задача | Мастер-промт (Notion) |
|---|---|
| Обычный продукт: интенсив, ISEEYOU, курс, маркетинговый МК, ОУ, массовое создание за месяц | [Создание продукта в Озме из Ноушн](https://www.notion.so/31f612c762af81ba999cc5657404528c) — `31f612c7…528c` |
| **МК по абонементу (АМК)** — список дат/времени, продукты `АМК{НН}-{ГГГГММ}`, `class_type=18`, `action_fin_project=28`, `price=2000`, без мастеров и занятий, одной транзакцией | [Создание МК по абонементу в Озме](https://www.notion.so/347612c762af814f9734fecb3a7eeea2) — `347612c7…eea2` |
| **Ссылка на оплату на сайте** — после создания продукта: клон услуги в Bitrix (iblock 16), заполнение полей, цены, обратная запись «Ссылка на оплату» в Notion | [Создание ссылки на оплату на сайте](https://www.notion.so/327612c762af81bb89fac8789b1e8f09) — `327612c7…8f09` |

> Признаки АМК: в задаче перечень строк вида «1 мая (пт) 13-16», тип «МК по абонементу» / «АМК». Каждая строка с датой = отдельный продукт. ⚠️ У АМК мастера НЕ добавляются и занятия НЕ создаются (нет мастера → триггер занятий не сработает) — это нормально, так и задумано.
> Признаки «ссылки на оплату»: просят «ссылку на оплату», дают ID продукта из таблицы Notion. Это работа на **сайте (Bitrix)**, а не в Озме — часть полей (MAIN_SEVICE, способы оплаты, лимит мест в торговом каталоге) проставляются **вручную в админке**, о чём предупреди пользователя.

## Критические правила (набиты практикой — в самом промте их нет)

### 1. Занятие создаёт триггер, не ты
Дочернее «Занятие» генерируется триггером `generate_lesson_after_master_adding` на `crm.masters_for_actions` — **при добавлении мастера**, а не при вставке продукта. Для одиночного однодневного продукта (даты `action_datetime` и `end_datetime` в один день) триггер создаёт ровно одно занятие сам.

**Поэтому порядок для одиночного МК/мероприятия:** (1) insert продукта → (2) insert мастера → занятие появится **само**. НЕ создавай занятие вручную — получится дубль, который придётся удалять. После — проверь:
`SELECT count(*) FROM crm.actions WHERE parent_action=<id> AND is_deleted=false` = ожидаемое число.
Для многодневных интенсивов генератор раскладывает занятия по дням недели из расписания — но всё равно сначала проверь, что натворил триггер, прежде чем досоздавать вручную.

### 2. Тип не из промта → уточняй у пользователя
Промт даёт фиксированные справочники для интенсивов, ISEEYOU и ОУ. Для **маркетинговых МК, курсов, мероприятий и прочего** значений нет. Не угадывай — найди в Озме 2–3 похожих продукта и предложи их конфиг, затем подтверди у пользователя `class_type`, `project`, `training_type`, `class_category`, `action_fin_project`, `fin_segment`.
Опорный конфиг **маркетингового МК** (Направление «Маркетинг»): `action_fin_project=36`, `project=1`, `class_category=9` (Другое), `fin_segment=1` (B2C), `class_type=2` (Мастер-класс) или `5` (Мероприятие), `training_type=5` (Творческий), `is_monitor_payments=false`.

### 3. Имя: код по промту vs описательное
Промт требует имя = CODE («Ссылка в CRM», напр. `MK01-202606`). Но команда иногда **вручную** переименовывает продукт в описательный формат «{Название для сайта} с {мастер в твор. падеже}» (напр. `veronikakorobova1` → «На языке дружбы с Александрой Сучковой»). Если в карточке «Ссылка в CRM» пустая и формат неочевиден — спроси пользователя, какое имя ставить. Кто и когда правил запись, видно через `query_events` (по `row_id`).

### 4. Время в UTC
OzmaDB хранит datetime в UTC. МСК = UTC+3 → перед записью **вычесть 3 часа** (`action_datetime`, `end_datetime`). Поля-даты `pl_start_date` / `pl_end_date` — без конвертации.

### 5. Подтверждение и идемпотентность
Перед вставкой — всегда сводка и подтверждение. Перед нумерацией — проверь занятые номера (`SELECT name FROM crm.actions WHERE name LIKE 'ПРЕФИКС%-ГГГГММ' AND type='Продукт'`). Не дублируй мастеров/заявки.

## Поиск ID в Озме

- **Мастер:** ФИО в `base.people` (`first_name`, `last_name`). Но `crm.masters_for_actions.contact` ссылается на `base.contacts` (общий id с `base.people`). При нескольких тёзках бери того, кто реально используется мастером:
  `SELECT contact, count(*) FROM crm.masters_for_actions WHERE contact IN (...) GROUP BY contact`.
- **Локация:** `crm.locations` = тип площадки (1 Москва, 2 Выездной, 3 СПб, 4 Онлайн, 5 Региональный) — НЕ адрес. Конкретный адрес из «Место проведения» кладём в текстовое поле `street`.

## Формат транзакции

```json
{ "operations": [ { "type": "insert", "entity": {"schema": "crm", "name": "actions"}, "fields": { } } ] }
```
> `entity` всегда объект `{"schema":"…","name":"…"}`, не строка.
