# ТЗ: Платформа агентов Гоголь Школы

**Статус:** черновик на согласование
**Дата:** 2026-06-18
**Автор:** Claude (по запросу Бэллы)

Этот документ описывает **общий каркас** для всех автономных/полуавтономных агентов школы. Конкретные агенты оформляются отдельными короткими ТЗ, которые ссылаются сюда. Первый частный случай — [Бот заведения продуктов](tz-bot-zavedenie-produktov.md).

---

## 1. Зачем платформа

Цель — переводить существующие скиллы (`/newproduct`, `/check-transactions`, `/new-refunds` и др.) в автономную работу: запуск по триггеру/расписанию, выполнение headless, проверка и правка людьми в Telegram. Двойная ценность: реальная автоматизация **и** опыт построения агентов.

Принцип проектирования: строим **не одноразовые боты, а переиспользуемый движок**, на котором каждый следующий агент дёшев.

---

## 2. Ключевая модель: движок + штат специалистов

Слово «агент» разложено на три слоя:

```
┌──────────────────────────────────────────────────────┐
│  ДВИЖОК (runtime)                  ← ОДИН на всё       │
│  сервер + Agent SDK + MCP + Telegram-loop + guardrails │
└──────────────────────────────────────────────────────┘
        │ на нём крутятся ↓
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ Агент       │ │ Агент       │ │ Агент       │   ← МНОГО,
│ «продукты»  │ │ «сверка»    │ │ «рассылки»  │     1 на воркфлоу
│ скилл+промпт│ │ скилл+промпт│ │ скилл+промпт│
│ +права+канал│ │ +права+канал│ │ +права+канал│
└─────────────┘ └─────────────┘ └─────────────┘
```

- **Движок (runtime)** — общая инфраструктура: как агент запускается, ходит в MCP, общается в Telegram, спрашивает подтверждение, логируется. Строится один раз (~80% работы первого агента).
- **Агент** = движок + **конкретный скилл + его промпт + его инструменты (MCP) + его канал + его guardrails**. Разные агенты = разная работа, разные права, разные триггеры, разные аудитории.

### Это «штат специалистов», а не «команда агентов»

| | Команда агентов (multi-agent) | Штат специалистов (наш случай) |
|---|---|---|
| Что делают | вместе решают **одну** задачу, переговариваясь | каждый — **свой** независимый процесс |
| Аналогия | мозговой штурм над одной проблемой | сотрудники разных отделов |
| Пример | «найди баги → проверь → синтезируй» | бухгалтер, продажник, маркетолог |

У нас процессы независимы (заведение ≠ сверка ≠ рассылки) → **штат специалистов**. Оркестрация под-агентов («команда») появится позже и только внутри *одной сложной* задачи (напр. «запуск продукта» = CRM + ссылка оплаты + расписание + анонс).

### Правило: 1 воркфлоу = 1 агент

Не делать один мега-агент «на всё»: разные guardrails (сверка читает — возвраты двигают деньги), раздутый промпт = больше ошибок, разные триггеры/аудитории, тяжелее чинить. Дёшево держать много мелких — движок общий.

## 3. Каналы (Telegram) и ретеншн

### Правило выбора канала

> **Группа = кто должен это видеть. Топик = какой конкретный случай.**

Права доступа в Telegram — на уровне **группы**, не топика. Поэтому **отдельная группа на домен/аудиторию**, топики — по экземплярам внутри.

### Паттерн под срок жизни элемента

Не каждый воркфлоу плодит топик на экземпляр. Выбор по «сроку жизни»:

- **Долгоживущее + разбор + запись** → **топик на экземпляр**, бот **закрывает** (`closeForumTopic`) по завершении; остаётся архивом.
- **Одноразовое, частое, выкидное** → **один постоянный топик-лента**, отчёты идут сообщениями; автоудаление старых.

### Таблица каналов и ретеншна

| Воркфлоу | Группа (аудитория) | Паттерн | Ретеншн / уборка |
|---|---|---|---|
| Заведение продуктов | «Заведение продуктов» (продукт-команда) | топик на продукт | закрыть после «Проверено» (+7 дн); удалять опц. через год |
| Сверка платежей | «Сверка платежей» (финансы) | **один топик-лента** | удалять сообщения старше 30 дн |
| Рассылки | «Рассылки» (маркетинг/SMM) | топик на кампанию | закрыть по завершении кампании |

> Если **все** процессы ведёт одна пара людей — допустимо одну группу с топиками по типам. Как только аудитории расходятся (финансы ≠ маркетинг) — разводить группы.

### Автоуборка

Боевой бот ходит в Telegram через **Bot API** (не через management-MCP), где есть полный набор форум-методов: `createForumTopic`, `closeForumTopic`, `reopenForumTopic`, `deleteForumTopic`, `deleteMessage`/`deleteMessages`.

Уборка = **scheduled-задача в n8n** (раз в сутки), ретеншн — настройка на группу, меняется без перестройки.

> Ограничение: наш management-MCP (`telegram_gogolschool`) форум-топиками, похоже, не управляет (есть `delete_message`, нет create/delete topic). Для прод-бота это неважно — он на Bot API.

## 4. Общие требования к любому агенту

Любой агент на платформе обязан:

1. **Тянуть мастер-промпт/логику из источника правды** (Notion / SKILL.md), не хардкодить.
2. **Confirm-first на запись** в боевые системы (Ozma/Bitrix), где это применимо — показать diff, ждать «да» (память `feedback_ozma_refunds`).
3. **Белый список операций** — агент работает только в рамках своего воркфлоу; всё за рамками — эскалация к человеку.
4. **Авторизация** — писать агенту-исполнителю могут только сотрудники из списка (по Telegram-ID), даже если в группе есть другие.
5. **Аудит** — каждый запуск и каждая правка логируются (n8n + обратная запись в источник).
6. **Идемпотентность** — повторный триггер не плодит дубли.
7. **Эскалация** — при неуверенности/выходе за скоуп: «передаю человеку», без импровизации на проде.
8. **Секреты** — токены MCP/Anthropic/бота в `.env`/секрет-хранилище, не в гите.

## 5. Инфраструктура (общая)

- **Сервер:** `ozma.gogol.school` (Timeweb, управляется командой). Всегда онлайн.
- **MCP по HTTP** уже развёрнуты (порты 8001/8002, токен-защита): Ozma, Bitrix. Notion — облачный MCP.
- **Ставим на сервер:** Node.js, n8n (оркестрация + scheduled-уборка), Claude Agent SDK (headless-движок), `git clone` репозитория со скиллами (`git pull` для обновлений).
- **Сеть:** эндпоинт n8n доступен из интернета для входящих триггеров (домен + HTTPS).
- **Доступ:** SSH к серверу есть под **`root`** (у Беллы). Root — **только для установки/обновлений**.
- **⚠️ Сервисный пользователь (безопасность):** агента (n8n + Agent SDK) запускать под **отдельным non-root юзером** (напр. `agentbot`) без sudo, с доступом только к: рабочей копии репо, `.env` с токенами, сетевым MCP-эндпоинтам. Автономный агент, пишущий в прод по свободному тексту из Telegram, под root недопустим — ограничиваем радиус поражения. Root оставляем администратору.

## 7. Этапность платформы

1. **Движок на продукт-боте** — построить первый агент так, чтобы инфраструктура (запуск, MCP, Telegram-loop, guardrails, уборка) была отделена от логики `/newproduct`.
2. **Извлечь каркас** — оформить общую часть переиспользуемой (конфиг агента: скилл + промпт + права + канал + ретеншн).
3. **Второй агент (сверка)** на том же каркасе — подтверждение, что новый агент = «конфиг + скилл», а не стройка с нуля.
4. **Фан-аут** — остальные агенты по карте.
