## 3. Что делает (поток)

### Фаза 0 — ветвление по площадке

Бот сначала смотрит поле **`Тип канала`**:
- **`mail`** → полный цикл ниже (Фазы A→B→C), включая отправку (статус+время).
- **`bot-*` / `tg-*` / `inst-*` / прочее** → бот заводит активность **только для истории** (Фаза A в части UTM/аналитики), **без отправки** — публикацию по этим каналам команда делает отдельно. Фаза B сводится к отчёту «заведено для истории», без шага «отправить».

### Фаза A — подготовка (по триггеру из карточки)

1. **Списки в CRM.** По полю `Сегмент` создать/привязать `marketing.lists` и наполнить `marketing.contacts_in_lists` контактами сегмента. Ссылку на список сохранить в **теле карточки**/комментарии (в Активности нет поля «Список» — см. §9.9).
2. **Шаблон в Unisender.** Склонировать нужный шаблон (через `unisender` MCP: `get_template` → `create_email_template`), подставив тему/текст/CTA из **тела карточки**. Получить `template_id`.
3. **UTM-ссылка.** Собрать UTM по промту (источник правды — prompt library в Notion; маппинг utm_source/medium/campaign из «Тип канала»/«Тип месседжа»/«Активность»/CTA). Записать в поле «UTM».
4. **Кампания в CRM.** Создать `marketing.campaigns`: `name`, `letter_theme`, `campaign_start_date`=Дата, `responsible_person`, `template_id`, `single_product`/`basket` (из «Наборы и проекты»), `stage`. Привязать списки через `marketing.lists_in_campaigns`.

### Фаза B — проверка и запуск (Telegram)

5. Создать **топик в Telegram-группе «Рассылки»** (топик на кампанию, см. платформенное ТЗ §3), запостить **отчёт** (что создано: списки + размер аудитории, шаблон, UTM, кампания, дата). Бот готов:
   - принимать **правки** (поменять сегмент/тему/UTM/дату) — через MCP, с confirm-first;
   - по явной команде ответственного **«отправить»** — запустить отправку (см. §6 про механику и §7 про guardrail).
6. После отправки — статус карточки `опубликовано`, в топик «ушло N адресатам в HH:MM».

### Фаза C — пост-аналитика (отложенно)

7. Через **~3 дня после публикации** (уточнено Беллой 2026-06-23) бот просыпается по каждой активности и собирает статистику (к этому сроку метрики устаканиваются; email/Метрика догоняют ~48 ч):
   - **Sent / Open Rate / Click Rate** — из Unisender MCP `get_campaign_stats` (и/или `marketing.email_statuses_for_actions`, event_type 3=прочитано, 4=клик — см. память `reference_email_engagement`);
   - **Payment** — выручка, атрибутированная кампании (через UTM → Я.Метрика `get_utm_report` / заказы/`fin`) — **правило атрибуции на уточнение, см. §8**.
8. Записать метрики в карточку **🎯 Активности** (поля §2 «Бот заполняет»), отчитаться срезом в топик Telegram, топик можно закрыть.

## 6. Механика отправки (разобрано)

**Отправка = смена статуса кампании в Ozma, а не прямой вызов Unisender.** Разведано по коду 2026-06-18:

- статусы `marketing.campaigns.stage` (`marketing.campaigns_statuses`): `draft` / `in_progress` / `completed` / `cancel` / `error`;
- триггер `update_fields_values_before_update_campaign_stage` (id 40) реагирует на смену статуса; механизм рассылки — `spreadAction` из модуля `campaigning.mjs`: «распространяет» мастер-действие кампании по контактам (создаёт `pm.actions` на каждый контакт из `contacts_in_campaigns` + списков `lists_in_campaigns`);
- строки «Unisender» в JS-коде Ozma **нет** — фактическую доставку письма по `template_id` выполняет **внешняя интеграция Ozma** (worker/outbox), боту не видна и им не управляется.

**Уточнено мастер-промтом кампании** ([367612c7…a9a7](https://app.notion.com/p/367612c762af8134bc68e3042613a9a7)): создание кампании — **неразрушающее**, бот создаёт `marketing.campaigns` в статусе **`Черновик` (id=4)** + задачу распространения в `pm.actions` (`from_campaign`, `type=Рассылка` id=3, `stage=Новое` id=5). А **отправка — два отдельных деструктивных шага**: «распространить действие» (spread по контактам) и «отправить рассылку». Промт их **не делает сам** — только с явного подтверждения.

**Для бота из этого следует (уточнено Беллой 2026-06-18):**
- **Подготовка** (фаза A) — безопасна, бот делает автономно: кампания в `Черновик` + задача распространения.
- **Отправка** (фаза B) — бот лишь **проставляет статус и время отправки**; всё остальное (распространение по контактам + фактическая доставка) делает **Ozma с интеграцией** по статусу и времени. Боту не нужно жать «распространить»/«отправить» и не нужен Unisender API — это не его слой. Важны два поля: **статус** + **время отправки** (`pm.actions.start_date`, МСК→UTC).
- Запуск отправки — только по **явной команде** человека в Telegram (см. §7).

> Внутренняя механика Ozma (какой именно статус, как срабатывает интеграция) — забота CRM, не бота. Бот выставляет статус+время по образцу рабочих кампаний; проверка результата — в фазе C (аналитика) и в Telegram-отчёте.

## 7. Guardrails (специфика рассылок)

Дополнительно к платформенным (`tz-agent-platform.md` §4):

1. **Отправка — необратимое внешнее действие** (письмо ушло тысячам, не отозвать). Запуск **только по явной команде** уполномоченного человека в Telegram, с показом: тема, размер аудитории, UTM, дата. Без авто-отправки по триггеру.
2. **Двойное подтверждение размера аудитории** — если список аномально большой/маленький относительно сегмента, бот предупреждает перед отправкой.
3. **Кто может жать «отправить»** — отдельный, более узкий список Telegram-ID, чем «кто может смотреть/комментировать».
4. **Тестовое письмо** — перед массовой отправкой бот предлагает отправить тест на адрес маркетолога.
5. **Идемпотентность** — повторный триггер по карточке со статусом `отправлено` не шлёт повторно.

## 8. Сбор статистики — источники

| Метрика | Источник | Статус |
|---|---|---|
| Sent | Unisender / кол-во `contacts_in_campaigns` | ✅ Unisender MCP `get_campaign_stats` (`sent`/`delivered`) |
| Open Rate | `marketing.email_statuses_for_actions` event_type=3 / Unisender | ✅ Unisender MCP `get_campaign_stats` (`open_rate_pct`) |
| Click Rate | `marketing.email_statuses_for_actions` event_type=4 / Unisender | ✅ Unisender MCP `get_campaign_stats` (`click_rate_pct`) |
| Payment | выручка по UTM → заказы/`fin`, либо по `basket` | Я.Метрика-UTM ✅ (цели/визиты), **правило атрибуции на уточнение** |

Маппинг «поле → источник» и нюансы (дробление списков >1000, несколько UTM) — в Notion [📊 Как вносить статистику в Активности](https://app.notion.com/p/388612c762af81619bcec028061e6b64).
