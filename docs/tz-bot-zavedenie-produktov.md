## 1. Цель

Сделать так, чтобы заведение нового продукта в CRM запускалось из Notion **без участия Беллы в Claude Code**, а ответственный сотрудник:

1. получал **уведомление в Telegram** с отчётом «что внесено»;
2. мог прямо в чате **проверить** результат (продукт, мастера, занятия, цена, ссылки);
3. мог написать боту замечание («стоимость не та», «мастер другой») — и бот **сам внёс правки в CRM**, ровно так, как сейчас это делает Claude в сессии (через те же MCP), с **подтверждением перед записью**.

По сути — воспроизвести нынешний UX Claude-Code-чата (см. как Белла работает со скиллом `/newproduct` сейчас), но для рядового сотрудника и в Telegram, с автозапуском по триггеру.

### Принятые решения

| Вопрос | Решение |
|---|---|
| Автономность правок | **«Как правит Claude»**: бот автономно правит через MCP, но перед каждой записью в Ozma показывает что собирается сделать и **спрашивает подтверждение** (правило `feedback_ozma_refunds` — confirm-first на запись в CRM). |
| Старт | Сначала это ТЗ → согласование → реализация. |
| Канал | Telegram (у школы уже развит Telegram-MCP: боты, контекст-сессии, автоответы). |

## 4. Сквозной поток (happy path)

1. Сотрудник в Notion выставляет карточке статус **«В очереди на CRM»** (вручную или кнопкой).
2. Notion-автоматизация шлёт **webhook** в n8n с `page_id` карточки.
3. n8n запускает **headless `/newproduct`** (Agent SDK) с этим `page_id`.
4. Агент отрабатывает скилл строго по мастер-промпту: создаёт продукт в `crm.actions`, мастеров, занятия (через триггер), цену/доли, обновляет «Ссылка в CRM» в Notion, делает самопроверку.
5. По завершении агент формирует **структурированный отчёт** и:
   - меняет статус карточки на **«Заведено, на проверке»**;
   - создаёт **новую тему (topic) в Telegram-группе** под этот продукт и постит туда отчёт; **контекст-сессия** агента привязана к этой теме (см. §7.1).
6. Любой из ответственных в группе читает отчёт. Если всё ок → пишет в теме «ок» → бот ставит статус **«Проверено»**.
7. Если есть замечание → пишет боту («стоимость 5000, мастер Петров»).
8. Бот в рамках контекст-сессии понимает, о каком продукте речь, формулирует правку, **показывает её и спрашивает подтверждение** («Меняю price 4000→5000 у продукта INT05-202606. Подтверждаешь?»).
9. После «да» бот вносит правку через MCP (как Claude), отчитывается «готово», обновляет отчёт.

### Что в отчёте (пример)

```
✅ Завёл продукт: «Интенсив по сторителлингу» (INT05-202606)
Тип: Интенсив · class_type 1 · fin_project …
Даты: 12–14 июля, 19:00–22:00 МСК
Мастер: Анна Иванова (contact 15741)
Локация: Москва · ул. Пример, 1
Занятий создано: 3 (проверено: триггер сделал ровно 3)
Цена: 12 000 ₽ · доли: 2
Ссылка в CRM: <url>
Ссылка на оплату: — (не создавалась)

Проверь. Напиши, если что-то поправить, или «ок».
```

## 5. Поведение бота «как Claude» — детально

Бот должен повторять рабочую модель Claude в сессии, а не быть «тупым исполнителем команд»:

1. **Тянет мастер-промпт из Notion** перед действием (как требует SKILL.md, шаг 1) — не хардкодит логику.
2. **Confirm-first на запись в Ozma** — перед любым `insert`/`update` в `crm.*`/`fin.*` показывает diff и ждёт «да» (память `feedback_ozma_refunds`).
3. **Соблюдает критичные правила скилла:**
   - занятие создаёт **триггер**, не агент (не плодить дубли — `reference_ozma_product_lesson_trigger`);
   - datetime в **UTC** (МСК−3);
   - неизвестный тип продукта → **не угадывать**, предложить конфиг похожих и спросить;
   - расписание/`detail_text` Bitrix — читать целиком перед записью, не затирать.
4. **Показывает что сделал** — отчёт с ID и ссылками, как сейчас в сессии.
5. **Держит контекст продукта** между репликами (контекст-сессия), чтобы правки адресовались верно.
6. **Не выходит за рамки продукта сессии** — «заведи ещё вот этот» = новый триггер/сессия, а не свободное блуждание по базе.

## 6. Guardrails (защита боевой базы)

Поскольку бот пишет в **прод** по свободному тексту из Telegram:

1. **Подтверждение перед каждой записью** в Ozma/Bitrix (обязательно).
2. **Белый список операций** — бот в режиме правки продукта может менять только поля этого продукта/его мастеров/занятий/цены и связанные сущности на сайте. Массовые `delete`, операции вне продукта — запрещены без эскалации к Белле.
3. **Доступ к боту** — только у авторизованных Telegram-ID сотрудников (маппинг сотрудник→права). Чужой не должен править CRM.
4. **Аудит** — каждый запуск и каждая правка логируются (n8n + комментарий в карточку Notion с кратким журналом). Кто/когда/что.
5. **Секреты на сервере** — токены MCP, ключи Anthropic, токен бота хранятся в `.env`/секрет-хранилище, не в гите.
6. **Стоп-слово / эскалация** — если сотрудник просит что-то за пределами скоупа или агент не уверен — пишет «передаю Белле», не импровизирует на проде.
7. **Идемпотентность** — повторный webhook по той же карточке не должен заводить дубль (проверка «Ссылка в CRM» уже заполнена / по статусу).

## 7. Триггер в Notion

**Предпочтительно:** нативная автоматизация Notion «при смене статуса на *В очереди на CRM* → Send webhook» (нужен тариф с webhook-действиями — **уточнить**, обычно Business/Enterprise).

**Запасной вариант** (если webhook-действия нет на тарифе): кнопка-свойство «🚀 Завести» с действием «Open link» на URL n8n-эндпоинта (`…/trigger?page_id=…`). Менее изящно, но работает на любом тарифе.

Статусная модель карточки (свойство Select/Status):
`Черновик → В очереди на CRM → Заведено, на проверке → Проверено` (+ `Ошибка` для сбоев).

### 7.1. Telegram: группа-форум с темами на продукт

Канал проверки — **не личка одному сотруднику, а группа-форум с темами (topics)**. У школы уже есть такой формат (напр. группа «Чат продажи GS (бот-тест)» с темами «Запрос LAB8», «Главный бот: консультация» и т.д.).

Модель:
- Заводится **отдельная группа** (или используется существующая) «Заведение продуктов», в неё добавлены **все ответственные** + бот.
- На каждый заведённый продукт бот **создаёт новую тему** (`topic`) с названием продукта (напр. «INT05-202606 · Интенсив по сторителлингу») и постит туда отчёт.
- **Контекст-сессия агента привязана к `chat_id + topic_id`** — поэтому любое сообщение в теме трактуется как реплика по этому продукту, без путаницы между продуктами.
- **Несколько ответственных** видят переписку и любой может включиться: написать замечание, подтвердить правку, сказать «ок». Бот отвечает в той же теме.
- Готовые/проверенные темы можно закрывать (`close topic`) — остаётся история.

Плюсы такого формата: коллективная проверка, прозрачность (всё видно команде), нет «бутылочного горлышка» из одного человека, история по каждому продукту в своей ветке. Авторизация (§6.3) проверяется по Telegram-ID автора реплики — писать боту-исполнителю могут только сотрудники из списка, даже если в группе есть кто-то ещё.

## 11. Риски и ограничения

- **Headless ≠ интерактив бесплатно.** Диалог правок требует постоянного агента с контекст-сессией — это сложнее разового запуска. Это самая трудоёмкая часть (этап 5).
- **Свободный текст → прод-база.** Снимается guardrail'ами (§6), но остаётся зоной внимания. Confirm-first обязателен.
- **Зависимость от мастер-промптов в Notion.** Если промпт сломан/изменён — бот отработает по нему же. Промпт остаётся единым источником логики (как и сейчас).
- **Стоимость токенов — незначительна.** При ~200 продуктов/год (~4/нед) запуски стоят копейки. Главная цена проекта — **разовая настройка + текущая поддержка** сервиса на проде, а не токены.
- **ROI при низком объёме.** 4 продукта в неделю — немного. Основная ценность бота не в экономии времени на заведении, а в **коллективной проверке силами сотрудников вместо Беллы** и автономности. Если ценность именно в этом — ок; если цель была «разгрузить по объёму» — при 200/год проще промежуточный вариант (кнопка-флаг + я завожу), и стоит это взвесить до стройки.
- **Поддержка.** Сервис на проде нужно мониторить (n8n-ошибки, упавшие MCP, протухшие токены).
